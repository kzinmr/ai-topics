# X Bookmarks Ingest Pipeline

Automated pipeline for ingesting X/Twitter bookmarks into the wiki. Runs as a cron job with a pre-run script.

## Architecture

Three complementary components:

| Component | Script | Purpose |
| --- | --- | --- |
| **Entity Skeleton Builder** | `build_x_wiki.py` | One-time: creates entity pages from X account profiles |
| **Bookmark Ingest** | `fetch_x_bookmarks.py` | Periodic: fetches new bookmarks, processes articles |
| **Account Posts Scan** | `fetch_x_accounts.py` | Periodic: fetches latest tweets from tracked accounts |

Do NOT merge or replace any of these — they serve different purposes.

## Pre-Run Script: `fetch_x_bookmarks.py`

Location: `~/ai-topics/scripts/fetch_x_bookmarks.py`

### Steps

1. **Load processed IDs**: Read `~/.hermes/processed_x_bookmarks.json` (contains `tweet_ids` array)
2. **Fetch bookmarks**: `xurl bookmarks -n 100 --auth oauth2`
3. **Dedup**: Filter out any tweets whose `id` is in the processed set
4. **Extract URLs**: For each new bookmark, collect external URLs (not x.com/twitter.com)
5. **Output JSON**: Print `{"new_bookmarks": [...]}` to stdout for the agent cron job
6. **Unbookmark processed**: For each new bookmark, run `xurl unbookmark <id> --auth oauth2` to remove from X's list
7. **Update DB**: Add all new tweet IDs to `~/.hermes/processed_x_bookmarks.json`

### Dedup Safety Net

Two layers prevent re-processing:
- **X-side**: `unbookmark` removes from the bookmark list, so next `bookmarks -n 100` won't return them
- **Local DB**: `processed_x_bookmarks.json` always updated — if unbookmark fails, local dedup still prevents re-processing

### DB Format

```json
{
  "tweet_ids": ["1234567890", "0987654321", ...],
  "unbookmark_failures": 0
}
```

- `tweet_ids`: sorted array of all processed tweet IDs (safety net)
- `unbookmark_failures`: count of unbookmark API failures (observability)

### Cron Setup

```yaml
job_id: <auto-generated>
name: x-bookmarks-ingest
schedule: 30 23 * * *   # daily at 23:30 UTC (08:30 JST next day)
script: fetch_x_bookmarks.py
deliver: discord:<channel_id>:<thread_id>
```

### Agent Prompt (from cron job)

The agent cron job receives the JSON output and:

1. **Score every bookmark against Karpathy's filter** (MANDATORY — do not skip). Classify into three tiers:
   - **Tier 1 (COMPOUND — 3x weight)**: Context engineering (long-context, retrieval, memory, prompt caching), Tool design (APIs/tooling for LLMs), Orchestrator-subagent pattern, Eval discipline, Harness mindset (harness > model), MCP, Practical production deployment. These get full wiki pages.
   - **Tier 2 (Relevant)**: Local model capability, LLM infrastructure, architectural patterns. These get raw article saves + optional quick patches to existing pages.
   - **Tier 3 (DEPRIORITIZE — skip/low weight)**: AutoGen/AG2, CrewAI, autonomous agent pitches, agent app stores, SWE-bench chasing, Microsoft Semantic Kernel, DSPy, viral frameworks, per-seat SaaS pricing. These get raw article saves only (no wiki pages).
2. **Fetch X native article bodies** for bookmarks with `x.com/i/article/...` URLs via `xurl -X GET "/2/tweets/<id>?tweet.fields=article,entities"` (see parse pitfall below)
3. **Scrape external URLs** (Reddit, blogs, etc.) via `web_extract`
4. **Save articles** to `~/wiki/raw/articles/YYYY-MM-DD_source--slug.md`
5. **Create/update wiki pages** for Tier 1 articles (concept pages, entity page updates). Tier 2: optional quick patches. Tier 3: raw save only.
6. **Update index.md and log.md**, then commit and push to `github.com/kzinmr/ai-topics`

### Batch-Scale Execution Pattern (10+ bookmarks)

When processing large batches, use `execute_code` for bulk operations instead of individual tool calls. A single Python script can:
- Create all new entity pages in one call
- Update all existing entity/concept pages in one call
- Handle index.md and log.md updates atomically

This reduces tool call overhead from ~30+ individual operations to ~3 `execute_code` calls, dramatically improving efficiency and reducing context window consumption.

**Typical large batch breakdown:**
1. `execute_code` — Save all article content files to `wiki/raw/articles/`
2. `execute_code` — Create all new wiki entity/concept pages + update existing ones
3. `execute_code` — Update index.md and log.md entries
4. `terminal` — git add/commit/push (single atomic commit)

## Common Issues

### Auth failures on unbookmark
- The `lucy` OAuth2 app must have bookmark write permissions
- Token expiry → X API returns auth error → unbookmark fails but DB still updates
- Rate limits → batch unbookmark one at a time with try/except

### Token expiration
- If `/opt/data/.xurl` token expires, both fetch and unbookmark fail
- The local DB is still updated, preventing re-processing
- **Partial token expiry**: bookmark.read scope may still work while read scope fails → cron returns `new_bookmarks: []` but user sees bookmarks in X UI
- **Diagnostic**: Always run `timeout 10 xurl --auth oauth2 whoami` first — if it requests browser auth, token is expired for all scopes
- Fix: user re-authenticates with `xurl --auth oauth2`

### Empty new_bookmarks
- Expected behavior after a successful run (all bookmarks unbookmarked + DB updated)
- If bookmarks are still appearing on X after unbookmark, the unbookmark API may have silently failed

### Large bookmark count
- `xurl bookmarks -n 100` returns up to 100
- If >100 new bookmarks accumulate between runs, some may not be unbookmarked
- Local DB prevents re-processing anyway

### xurl "No apps registered" — complete auth break
- Symptom: `xurl auth status` returns "No apps registered. Use 'xurl auth apps add' to register one."
- Effect: ALL xurl commands fail (bookmarks, user lookups, etc.) → cron returns empty `new_bookmarks: []`
- Root cause: OAuth2 app was never registered, or credentials were wiped
- Fix: User must run `xurl auth apps add <name> --client-id ... --client-secret ...` then `xurl auth oauth2 --app <name>` then `xurl auth default <name>`
- Debugging: If cron returns empty but user sees bookmarks in X UI, first check `xurl auth status` — this is the #1 cause

### Bookmarks with external_urls: [] — X-native content
- Some bookmarks contain only X-native content (X article cards, images, videos, native replies)
- The script correctly extracts `external_urls` but finds none → bookmark ID is still processed and unbookmarked
- No wiki article is created because there's no external URL to scrape

**X article content via xurl (REST endpoint)** — Bearer token with `read-write-directmessages` access:
```bash
xurl -X GET "/2/tweets/<tweet_id>?tweet.fields=article,entities"
```
- Returns JSON with `data.article` containing: `title`, `plain_text` (full body), `preview_text`, `cover_media`
- `plain_text` は **記事の全文** — 必ず処理すること。見落とし厳禁。
- `entities` はtweetレベル（外部リンクの抽出に使用）
- ⚠️ `-v` フラグは使わない（HTTPヘッダーがstdoutに出力されJSONパースが破壊される）

**Workflow for X native articles (primary method):**
1. Extract `tweet_id` from bookmark
2. Run `xurl -X GET "/2/tweets/<tweet_id>?tweet.fields=article,entities"` (no `-v` flag!)
3. Parse `data.article.plain_text` — **this is the full article body**
4. Save to `~/wiki/raw/articles/<tweet_id>_<slug>.md`
5. Create/update wiki entity or concept pages
6. Commit and push to `github.com/kzinmr/ai-topics`

**⚠️ CRITICAL:** Always process `plain_text` — it contains the full article body. Never skip it.
