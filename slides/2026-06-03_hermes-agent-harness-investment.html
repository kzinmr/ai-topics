Part 2 ・ The Bet

賭ける
— ハーネスとインフラ

投資対象 ・ 1/3

Open Harness と Framework は別物

// 使う面（人間が AI を使う）

Open Harness

Hermes / OpenClaw / OpenCode / Pi

  • CLI・チャット・ファイル・シェル・MCP
  • 広い柔軟性（使う柔軟性）
  • operator safety
// 組み込む面（AIをシステムに）

Framework / Runtime

Agents SDK / LangGraph / Pydantic AI

  • typed state・graph・durable exec
  • 深い柔軟性（作る・運用する）
  • product / tenant safety

workflow 抽象は薄れても、runtime 抽象（state/permission/memory/policy）は重要化する。

投資対象 ・ 2/3

「ハーネスを捨てても業務ロジックが残る」構成

注意：「モデルはハーネスに合わせて post-train される」→ メモリ/スキルは非可搬。堀は同時にロックインでもある。

投資対象 ・ 3/3

ハーネスは消えない、移動する

良いモデルは古い足場を不要にするが、新しい難所と失敗モードを解禁する。だから一度きりの投資ではない。

  • 検証は出力側に（テスト・閾値・自己硬化）
  • コンテキストは希少資源（compaction / offload / 目次化）
  • Ratchet で足し、腐ったら消す
Hosting ・ $5 → AWS

安く始め、痛みで上げる

brain / hands split：推論（脳＝gateway, 非公開・秘密情報）と コード実行（手＝sandbox, 最小権限）を分ける。エンドポイントを開くたび攻撃面が増える。

Economics

トークンではなく、結果で測る

per token → per outcome

「トークンの請求書は、不安定な量の仕事を表す、安定した単位のコスト」。同じ処理でも 5〜10倍ぶれる。

MiniMax は Gemini より per-token は高いのに、成功タスクあたり 2.3倍安い。トークン価格は選定段階で誤導する。

…と、その逆風

2026/5 後半：AI の ROI は「正当化しづらい」（Uber COO）。FT は大手 ROI をマイナス試算。

結果への帰属が要るのは、生のトークン投入が確実にリターンへ変換できていないから。出力が本当に判断を変えているか、を問い続ける。

前編の個人の脳は一番下の小さな実装。Control Room も Ratchet も監査可能な git も、縮尺が違うだけ。

Takeaways ・ 後編

開発者への5つの持ち帰り

  1. ハーネスは薄く、知識資産は厚く、可搬に。 中核はハーネスの外（DB・git）へ。
  2. ハーネスは消えず、移動する。 足し算(Ratchet)と引き算(削除)の両方。
  3. 安く始め、痛みで上げる。 $5 VPS → サンドボックス分離 → AgentCore。
  4. 増やすときは型に従う。 Control Room、ブレイン層、そして育てる。
  5. 個人と組織は相似形。 複利のファイルが堀、per outcome で測る。
End of Part 2 — 投資編

道具は乗り換えてよい。
ファイルは手放すな。

複利で育つあなた自身のファイル群だけは、ベンダにもハーネスにも預けず、自分の手元（git）に持ち続けること。それが、モデルがどれだけ強くなっても陳腐化しない唯一の資産だ。
