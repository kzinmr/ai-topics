02
Part 2 ・ Cron

Cron パイプライン
— 28ジョブを「構造」で捉える

3 Modescron は LLM だけでなく script / skill も登録できる

cron に何を登録できるか — 3モード

ジョブの大半は prompt + script + skills のハイブリッド。skill モードは _overrides/_custom（git）を外部呼び出し → 公式同梱skill🟩と棲み分け。

Pre-run ・ wake-gatescript は事後ではなく事前に走る

cron × script：決定的な実行 / コンテキスト収集

必要な時だけ高価な LLM を起こす。no_agent なら script が全て（stdout を verbatim 配信、LLM 不在）。例：sitemap-monitor / pipeline-watchdog。

workdir

cron × AGENTS.md — workdir でポリシーを一元管理

グローバルポリシーが、全 wiki 系 cron に届く

ai-topics の運用規約：全 wiki 書き込み系 cron は workdir 必須（未設定なら AGENTS.md は読まれない）。

Evolution

cron の進化 — read-only から自動適用へ

安全のため read-only から始め、確定的な action だけ自動適用へ昇格。最終ゲートは pre-commit（Part 1）。自動化の Ratchet。

Catalog ・ 1/2

ジョブカタログ① — ingest（受動）/ crawl（能動）

// 📥 ingest（passive）— 来たものを取り込む
  • blog-ingest → triage → wiki-ingest（RSS × blogwatcher cli）
  • newsletter-ingest → … → wiki-ingest（Substack 由来）
  • x-bookmarks-ingest（手で追加した新着 bkm を取得）
// 🔭 crawl（active）— 自分から取りに行く
  • active-crawl（hot-topics.yaml ベースの探索）
  • trending-topics（トレンドリサーチ）
  • sitemap-monitor（RSS 非対応サイトの監視）
  • x-accounts-scan（x-accounts.yaml の新着投稿）

各段は script(pre-run)＋skill のハイブリッド。X API は従量課金のため x-accounts-scan は約10アカウントに限定（日 <$1）。

Catalog ・ 2/2

ジョブカタログ② — lint（健全性）/ report（配信）

// 🩺 lint（health-check）— 自己修復
  • wiki-health-fix（スキャン→自動修正）
  • wiki-watchdog-fix（修正後の検証）
  • wiki-graph-analysis（重複・リンク）
  • tag-audit-weekly / check-skill-inventory

// 📣 report（delivery）— 向こうから届く

03
Part 3 ・ Skills

スキル — 経験をコードにする
公式を土台に、自分で育てる

SkillSKILL.md（YAML frontmatter, agentskills.io 互換🟩）

スキルとは — 「手順メモリ」

// ~/.hermes/skills/<name>/SKILL.md
---
name: llm-wiki
version: 2.0.0
license: MIT
---
# When This Skill Activates
# Core Operations
# Pitfalls …
要素内容
トリガー条件いつ使うか
手順具体的コマンド・パス
落とし穴失敗と対処
検証ステップ成功の確認

タスク時に関連スキルを自動ロード（progressive disclosure）。失敗を手順に変換し、二度と繰り返さない。

3 separate mechanisms

自律拡張・整理・回収 — 主体もトリガーも別

スキルの生成・修正は Hermes Agent 本体（セッション内）。背景の Curator や cron とは別物。

トリガーも主体も別。 ①はHermes自身、②は公式の背景GC、③はカスタムcron。

3 Layersカスタムには ①使用 と ②更新耐性backup の2役割

公式 × カスタム — スキルの3層構成

cron が叩く層＝決定的に使う（_custom/_overrides）、_adhoc・公式＝自律的に使う。git管理で更新でも消えない。

Extension公式の skill機構🟩 の上に、2通りで足す

拡張の2チャネル：_overrides と _custom

// _overrides/ — 公式skillをドメイン向けに上書き　🟧→🟩
llm-wikiblog-post-writingcron-job-managementwiki-graph-healthresearch-paper-writing

公式skillの挙動を、この知識ベースの作法に合わせて差し替える。

// _custom/ — 完全独自のドメインskill　🟧
dreamingdaily-rss-triagesemantic-article-groupingwiki-ingestion-pipelinesgrokipedia-enrichment

公式に無いパイプラインを丸ごと新設する。

_overrides は公式を上書き＝drift管理が要る、_custom は新規＝自由。どちらも cron が名前で参照する。

Update ・ drift

Hermes 更新と skill drift — バックアップが効くのは _overrides

git backup の本当の理由：更新で公式が変わっても保全するため（manifest + origin-hash の3-way merge）。

Case ・ llm-wiki

実例: llm-wiki — 公式を分析し、ドメインに適合

公式の骨格はそのまま、このドメインの作法（収集・採点・英語wiki化）を上乗せした。

04
Part 4 ・ Workflow

日々のワークフロー・活用方法
狭義RAG → 広義RAG

狭義RAG → 広義RAGwiki は「文脈の基盤」。すべてオンデマンド

活用の全体像 — 取り込み→クエリだけではない

「検索して答える」だけでなく、蓄積した文脈で生成を駆動する。この層は「自動の土台」（Part 2）の上に乗る。

CLI / MCP

外部エージェントから wiki を使う — CLI と MCP

どちらも wiki を壊さない（read-only）。CLI＝shell が使える agent 向き、MCP＝MCP クライアント向き。併用可。

On-demand ・ 文脈注入

ナレッジクエリ — 文脈注入の例

例：「GRPO の論文と実装の関係を整理して」「先週の ○○ 周りの議論をまとめて」

  1. wiki/index.md から関連ページを特定（concepts/ entities/）
  2. 必要なら raw/ のソースまで遡る
  3. 回答を返す ／ 再利用可能なら wiki/queries/ に保存

回答が一回で消えない。Wiki ページとして残り次回は即参照 —— Karpathy の compiled knowledge。RAG の「都度再発見」とは違う。

On-demand ・ 入力

オンデマンド取り込み（広義の入力）

例：「この記事読んで wiki に追加して」＋URL ／ 「この一連の議論を概念ページに」

  1. ソースを raw/ に保存（不変）
  2. index.md で既存ページを確認 → 更新 or 新規
  3. frontmatter・最低2つの [[wikilink]]・index.md/log.md を整備
  4. pre-commit を通して git commit & push

人間の介入：URL や指示を渡すだけ。取り込みも生成も、同じ wiki を介してつながる。

広義RAG ・ 出力

wiki から生成する — 聴衆・媒体別の reporting / blog / slides

// reporting
wiki-daily-reporttrending-topics-reportinghermes-report-quality

Gwern 品質（Anti-Examples 等）で、Slack / Discord / Telegram へ媒体別に整形。

// blog / slides
blog-writingresearch-paper-writinghumanizerclaude-design

humanizer で機械臭を除去。媒体別デザインで仕上げる。

同じ知識ベースから、聴衆（社内/一般）・媒体（chat/blog/slides）別に出し分ける。 ※この資料も wiki から生成。

Config

設定は対話的に・人間レビュー可能に

設定変更は対話で修正して運用。設定は保守でき・人間がレビューできる形（Markdown/YAML + git）。

// 対話的な設定修正（オンデマンド）
  • 「追跡対象に @x を追加して」→ config/feeds/*.yaml 編集 → スケルトン生成
  • Cron 失敗 → ログ分析 → スクリプト修正 → git commit
  • 人間の介入は設計判断のときだけ
// 失敗は改善資産に（軽く）
  • 繰り返した失敗 → harness 側へ恒久対策（スキル / pre-commit / AGENTS.md）
  • 「mistake → engineer a solution → never again」（Hashimoto）

すべて diff が残る。設定もスキルも cron も、git で監査・巻き戻しできる。

Wrap-up

まとめ — 「どう知識を扱うか」

半年運用して変わったのは 「何を知っているか」ではなく「どう知識を扱うか」。知識ベースは静的repoでなく、自律的に更新され続ける runtime 環境になった。

// 開発者への示唆
  • 設定は Markdown/YAML + git（人間もレビュー可）
  • 失敗は git に残し harness で恒久対策
  • スキルは自動で生まれ、自分で育てる
  • 定期実行を前提に設計（常駐）
  • 検証しやすい Environment を選ぶ
// いまの規模
  • Wiki 2,327 ページ（concepts 1,550 / entities 751）
  • Cron 28（26 active）／ スキル 100+
  • 追跡 RSS 84 ／ X 30+

LLM の外側の「設定・スキル・定期実行」こそ、モデルを仕事に変える境界。狭義RAG から広義RAG へ。
