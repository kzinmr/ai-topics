# 分割と双対——マルチエージェント、RLM、そして情報検索を貫くひとつの構造

Anthropicが2025年6月に公開した「How We Built Our Multi-Agent Research System」を読んだとき、まず目を引いたのは数字だった。Claude Researchのマルチエージェントシステムは、単一のClaude Opus 4を**90.2%**上回る——単なる改善ではなく、別物と言っていい差だ。

しかし、この記事の真価は数字ではない。彼らがなぜそこまで詳細にアーキテクチャを開示したかといえば、マルチエージェントを動かすには単に「エージェントを増やす」以上の設計思想が必要だったからだ。そしてその設計思想を深く読んでいくと、奇妙な既視感に襲われる。この構造、どこかで見たことがある。

RLM（Recursive Language Models）だ。そして情報検索（IR）のクエリ-文書双対性だ。

この記事では、一見バラバラに見える3つの技術——マルチエージェント、RLM、古典的IR——を貫く、ひとつの構造を描き出す。

## 1. 分割統治——なぜ「分ける」ことが効くのか

まずは地面を固めよう。マルチエージェントとRLM、この2つは外見がまったく違う。しかし、解決しようとしている根本問題は同じだ。

**LLMにはコンテキストウィンドウという容量制限がある。**

一度に処理できるトークン数には上限があり、それを超える入力を与えることはできない。では、上限を超える情報を扱いたいとき、どうするか？

### マルチエージェント：タスクを水平に分割する

AnthropicのClaude Researchは、リサーチタスクそのものを分割する。リードエージェント（Opus 4）がユーザーの質問を受け取り、「S&P 500 IT企業の全取締役を特定せよ」といった複合的なタスクを、企業ごと、セクターごとのサブタスクに分解する。それぞれのサブタスクは独立したサブエージェント（Sonnet 4）に割り当てられ、各サブエージェントは自分のコンテキストウィンドウの中で検索し、考え、結果をまとめる。最後にリードがそれらを集約する。

これはMapReduceだ。タスク空間を並列化し、それぞれの部分結果を独立に計算してから統合する。分割の方向は**水平**である。

### RLM：入力を深さ方向に分割する

一方、Alex Zhang、Tim Kraska、Omar Khattabらが提案したRLM（Recursive Language Models）は、入力そのものを分割する。10万トークンの文書を一度に読めないなら、チャンクに分けて再帰的に処理すればいい。モデルは自分自身を関数として呼び出し、部分結果を合成しながら、最終的に全体を理解する。

これは分割の方向が**深さ**（再帰的）である点を除けば、同じ発想だ。どちらも「単一のコンテキストウィンドウでは処理できない」という制約を、分割統治（divide and conquer）で乗り越えている。

| 次元 | マルチエージェント | RLM |
|---|---|---|
| 分割するもの | タスク | 入力 |
| 分割の方向 | 水平（並列） | 深さ（再帰） |
| 集約方法 | リードが統合 | 再帰呼出し結果を合成 |
| 得られるもの | 15×のトークンで90.2%向上 | 10M+トークン処理で精度劣化なし |

ここまでは比較的わかりやすい。問題はここからだ。

## 2. クエリ-文書双対性——エージェントと文脈の隠れた関係

マルチエージェントの内部をもう一段深く見ると、より subtle な構造が現れる。サブエージェントに与えられる「タスク指示（objective）」と、サブエージェントが返す「探索結果（context）」の関係だ。

これは、情報検索（IR）の分野で半世紀にわたって研究されてきた**クエリ-文書双対性**と相同である。

IRの世界では、クエリ（検索語）と文書（検索結果）は本来別物だが、同じベクトル空間に射影できるという発見があった。クエリは「短い文書」であり、文書は「長いクエリへの回答」である。この双対性はpseudo-relevance feedbackという技法に結実する——検索結果の上位文書を使ってクエリを拡張し、再度検索することで精度を上げる。クエリと文書の境界が、反復によって融解していく。

同じことがエージェントシステムでも起きている。

サブエージェントへのタスク指示（「この企業の取締役会メンバーを全員特定せよ」）は、IRのクエリに相当する。サブエージェントが探索して返す結果は、IRの文書に相当する。そしてサブエージェントが結果を評価し、必要ならさらに深掘りする「interleaved thinking」は、IRのrelevance feedbackそのものだ。さらにその結果を受けてリードエージェントが「もっとこの方向を調べろ」と新しいタスクを発行するのは、クエリ拡張である。

| IR | マルチエージェント |
|---|---|
| クエリ = 情報要求 | タスク指示(objective) = 調査指示 |
| 文書 = クエリへの回答 | 探索結果(context) = タスクへの回答 |
| 同じ意味空間に射影 | 同じトークン空間で表現 |
| relevance feedback | interleaved thinking |
| クエリ拡張 | objective更新 |

この双対性の核心はこうだ。「タスク指示」と「探索結果」は、本質的に**同じ意味的対象の異なる粒度での表現**であり、どちらもLLMのトークン空間に射影される。サブエージェントはタスク（クエリ）をコンテキスト（文書）に展開し、リードエージェントは文書を圧縮して新たなクエリを生成する。

Anthropicが「search is compression」と言ったとき、彼らはこの双対ループを指していたのだ。文書空間での探索結果を、クエリ空間での判断に圧縮する——IRがクエリ→文書→適合フィードバック→クエリ拡張と回すのと同じループを、エージェントはobjective→探索→interleaved thinking→objective更新として回している。

## 3. なぜこの構造に気づくことが重要なのか

技術的な類似を見つけて「面白いですね」で終わらせるのは易しい。しかし、この構造認識には実践的な含意がある。

### 第一に、エージェント設計の定石が変わる

もしマルチエージェントの協調がIRのクエリ-文書フィードバックループと相同なら、IRの分野で50年かけて蓄積された知見——クエリ拡張、適合フィードバック、ランキング融合、多様性確保——が、エージェント設計に直接転用できる可能性がある。たとえばAnthropicが「tool description」の最適化に苦労したように、IRの「クエリ表現の最適化」の知見はエージェントのタスク指示設計に応用できる。

### 第二に、分割の軸は交換可能である

マルチエージェント（水平分割）とRLM（深さ分割）は、同じ問題への別解だとわかれば、両者を組み合わせる発想が自然に出てくる。実際、RLMの論文では「Tool-Augmented RLM」として、再帰処理の各ステップでツールを呼び出す拡張が提案されている。これは水平と深さのハイブリッドだ。

### 第三に、そして最も重要なこと——「情報要求」と「情報充足」の境界が溶けつつある

LLMの時代において、プロンプトと補完、タスクと結果、クエリと文書——これらはすべて同じトークン空間上のグラデーションでしかない。かつて情報検索は「クエリ」と「文書」を別物として扱うことから始まったが、LLMは両者をひとつの連続体として扱う。タスクを書くことも、結果を受け取ることも、同じ言語モデルが同じ空間で行う。

このことは、エージェントの「自律性」についての議論を根底から変える。人間が与えるタスク指示（objective）と、エージェントが自律的に生成する探索結果（context）は、本質的に同じものの異なる解像度に過ぎないのなら、「自律性」とは単に解像度の勾配をどこまで下るかの問題になる。Anthropicがプロンプトエンジニアリングを「最重要レバー」と呼んだのも、この勾配の設計こそがエージェントの振る舞いを決めるからだ。

## 4. 着地点——「分ける」ことと「つなぐ」こと

ここまで書いてきたことを一言でまとめると、こうなる。

**LLMの能力を制約するのはコンテキストウィンドウの有限性だが、その制約を突破する方法は「分ける」ことにある。そして「分けた」ものを再び「つなぐ」とき、そこにはクエリと文書、タスクと結果、要求と充足の双対ループが現れる。**

この構造は、おそらくLLMに限った話ではない。容量制約を持つあらゆる情報処理システムが、制約を超えてスケールしようとするとき、分割統治と双対フィードバックという同じパターンに収斂していくのかもしれない。

マルチエージェントとRLMは、その最初の顕著な現れだ。これから数年で、このパターンの変種がさらに多く出現するだろう。それらを見つけたとき、「ああ、またこれか」と思えることが、おそらく設計者としての第一歩になる。
