# FDEかSaaSか、ではない——AI Agent時代に最も価値のあるポジションはその境界にある

「SaaSは終わる。FDEへ行くべきだ」。そう言う人は増えている。OpenAIは2026年5月にDeployment Companyを立ち上げ、150人のForward Deployed Engineer（FDE）を顧客組織に送り込む体制を整えた。AnthropicもBlackstoneやGoldman Sachsと組み、中堅企業向けのAI導入サービス会社を発表した。GoogleはGemini Enterprise Agent PlatformでAgent統治基盤を作りつつ、FDEがSIerと組んで現場に入る。

一見すると、SaaSプロダクトを作るより、FDEとして顧客現場に入ったほうが価値が高いように見える。しかし、この「FDE vs SaaS」という二分法そのものが間違っている。

**最も価値のあるポジションは、FDEでもSaaSでもない。その境界にある。**

この記事では、AI Agent時代にSaaSがどう構造変化し、どのような人材が勝ち、どのようなSaaS企業が生き残るのかを、2026年5月時点の複数ソースの分析から描く。

## 1. SaaSは終わらない。変わるのは「価値の単位」だ

まず、SaaSが死ぬという見方は誤りである。SaaSは**消滅するのではなく、構造が根本から変わる**。

旧来のSaaSは「全顧客共通の画面・機能・ワークフロー・レポートを、なるべく汎用的に作る」ことが中心だった。これからのSaaSは「顧客ごとの業務・データ・権限・評価基準・例外処理に接続し、AI Agentが実際に仕事を完了する状態を作る」ことが中心になる。

つまり、**機能配布プラットフォームから、Agent運用OSへの転換**だ。

この転換をレイヤー別に整理すると、次のようになる。

| レイヤー | 旧来SaaS | Agent時代のSaaS |
|---|---|---|
| **UI** | 全顧客共通の画面 | 薄いUI。Slack/Teams/メール/Agent UIが入口に |
| **機能** | 共通機能の追加 | Agentがツールを呼んで業務を遂行 |
| **カスタマイズ** | 管理画面、設定、ワークフロー定義 | メモリ、スキル、ポリシー、MCP、評価セット、承認フロー |
| **マルチテナンシー** | DB/計算資源の共有分離 | Agent実行・権限・監査・コスト・評価・メモリのテナント分離 |
| **価値単位** | 席数、ログイン、画面操作 | 完了タスク、削減時間、解決件数、品質、リスク低減 |
| **競争優位** | 機能数、UX、データ蓄積 | 業務データ、権限モデル、評価、監査、統合、導入知見 |

ここで重要なのは、**SaaSのマルチテナンシーが消えるわけではない**という点だ。むしろ、テナンシーの対象が「DBの行」から「Agentの実行環境」へと拡張される。Agent実行、権限、監査、コスト、評価、メモリ——これらすべてをテナントごとに分離・管理する必要がある。旧来SaaSより、むしろ設計負荷は高い。

## 2. Agentが増えるほど、Agentを統治するSaaSが必要になる

一見すると逆説的に聞こえるが、Agentの爆発的普及は**SaaSの需要をむしろ増大させる**。理由は単純だ。Agentは自律的に動くほど、Identity、Permission、Audit Log、Rollback、Human Approval、Policy Enforcementといった統治の仕組みを必要とする。

この「Agent統治レイヤー」を、我々は **Agent Control Plane（エージェント制御基盤）** と呼ぶ。以下の13のコアコンポーネントからなる。

| コンポーネント | 機能 |
|---|---|
| Agent Registry | デプロイされた全Agentの一元在庫管理 |
| Agent Identity | 非人間アイデンティティの管理 |
| Permission Management | 最小権限、委任権限、ツールレベルアクセス制御 |
| Execution Logs | Agentの全アクションの監査証跡 |
| Tool Call History | ツール呼び出しの履歴管理 |
| Cost Management | Agent単位・テナント単位のコスト追跡と上限設定 |
| Evaluation Results | 継続的な品質測定 |
| Failure Classification | 失敗の分類と自動ルーティング |
| Human Approval | 高リスク操作の条件付き承認ゲート |
| Rollback | 安全な取り消し・巻き戻し |
| Tenant-Specific Memory | テナントごとに隔離された業務メモリ |
| Security Policy | プロンプトインジェクション防御、ツール権限制御 |
| Audit Export | コンプライアンス対応の監査エクスポート |

これが偶然の一致でないことは、主要プレイヤーの動きを見れば明らかだ。

- **ServiceNow**：自社プラットフォームをAgentの「control tower」と位置づけ、Agent Orchestratorで複数Agentを部門横断で協調させる
- **Workday**：Agent System of Recordを発表し、AI Agentを人間の従業員と同様にライフサイクル管理する方向へ
- **Google Cloud**：Gemini Enterprise Agent Platformの中核に、Agent Identity、Agent Registry、Agent Gateway、Evaluation、Observabilityを据えている
- **Deloitte**：2026年にSaaS経由のAI Agent利用が急増し、「control centers」やAgent marketplaceが重要になると予測

「AgentがSaaSを壊す」のではない。**Agentが増えるほど、Agentを登録・統制・監査・評価するSaaSが不可欠になる**という、もう一つの力が働いている。このベクトルを見落とすと、FDEバブルの裏側で進行している巨大なプラットフォーム市場を見逃すことになる。

## 3. Tenant Agent Pack——「顧客ごとの機能は作らない」を守りながら顧客適応する方法

SaaSの保守性の要請は今後も変わらない。「顧客ごとにコードを分岐させるな」は鉄則として残り続ける。

しかし、Agent時代には顧客ごとの差異を完全に回避することは不可能だ。業務フロー、例外処理、用語、評価基準、承認条件——これらは顧客によって本質的に異なる。旧来SaaSはこれを「設定項目」や「機能追加」で吸収しようとしたが、そこに限界があった。

代わりに何をするか。**差異を置くレイヤーを変える**。コードではなく、Agentの「運用アーティファクト」の層で差異を管理する。これを **Tenant Agent Pack** と呼ぶ。

Tenant Agent Pack に含まれるものは以下である。

- 業務目的・Agentロール定義
- 利用可能/禁止ツールのallowlist/blocklist
- 承認が必要な操作一覧
- 参照可能なデータソース
- 顧客固有の用語集
- 例外処理ルール
- 評価データセット（その顧客にとって「良い」とは何か）
- 成功KPI
- エスカレーション先と条件
- コスト上限
- ロールバック手順
- プロンプト/システム指示
- 実行ログ保持ポリシー

これが意味するのは、「顧客ごとの機能を作らない」というSaaS原則が**守る対象を変える**ということだ。旧来は「コードを分岐させない」だった。これからは「コアのAgentエンジンを分岐させない」であり、その上に乗る運用アーティファクトは顧客ごとにバージョニングされ、テンプレート化され、再利用される。

この設計ができるSaaS企業は、FDE的な深い導入と、SaaS的な再利用性を両立できる。

## 4. FDEは「個別開発者」ではない——現場で発見し、プロダクトに抽象化する人だ

ここまでSaaS側の構造変化を見てきた。では、FDEとは何者なのか。

OpenAIのFDE求人要項にはこう書かれている。「実用的なパターンをツール・プレイブック・ビルディングブロックとして体系化（codify）する」。AnthropicのFDE求人要項には「再現可能なデプロイメントパターンを特定・体系化し、Product/Engineeringへフィードバックする」とある。

つまり、一流のFDEは「個別顧客向けに何でも作れる人」ではない。**現場でしか見えない要件を発見し、再利用可能な抽象へ変換する人**だ。

ここに、FDEとSaaSの境界で最も価値が高いポジションが生まれる。FDEで得た知見を、一回限りの納品で終わらせるのではなく、Agent Pack、評価基盤、権限モデル、Control Plane、業界テンプレートとしてプロダクト化できる人材——**Forward Deployed Product Engineer**と呼ぶべき存在だ。

この人材の価値は以下の3層で発揮される。

1. **現場層**：顧客の実際の業務を観察し、人間が何を見て判断し、どの画面を跨ぎ、どこでExcelが挟まり、何を怖がっているかを理解する
2. **抽象化層**：「この承認フローは実は業界共通ではないか」「この例外処理はPolicy DSLにできないか」「この評価セットはテンプレート化できないか」と見抜く
3. **プロダクト層**：発見したパターンを、Tenant Agent Pack、Agent Control Planeのコンポーネント、評価フレームワークとして実装する

純粋FDE（現場に張り付き、プロダクト構築に時間を割けない）でも、旧来型SaaS開発者（画面と機能だけを作る）でもない。この境界に立つ人材が、最も希少で価値が高い。

## 5. どのSaaSが死に、どのSaaSが生き残るか

以上の構造変化を踏まえると、SaaS企業の勝ち負けはかなり明快に分類できる。

### 危険なSaaS

**画面＋DBだけのSaaS**。入力フォーム、承認フロー、ダッシュボードで構成され、AgentがAPIやブラウザ操作で容易に代替できるもの。ユーザーが画面を開いて入力・検索・転記するだけの価値は、AI Agentに吸収される。

**薄いAIラッパーSaaS**。LLM APIにUIを被せただけ、プロンプトテンプレートを売るだけ、汎用チャットを業界別に言い換えただけの会社。モデルベンダー、個人Agentハーネス、既存SaaSのAgent機能の三方向から挟まれる。

**水平SaaSで業務状態を持たないもの**。データ・権限・業務文脈を顧客側に依存していると、Agent時代には「どの業務状態に対して、誰の権限で、何を実行し、どう監査するか」という価値の軸から外れる。

### 強いSaaS

**System of Recordを持つSaaS**。CRM、CS、会計、法務、HR、ITSM、物流、医療、金融、製造、建設——業務の現実がデータとして蓄積されている領域だ。Agentが実際に仕事をするためには信頼できるデータ・権限・履歴・状態管理が必要であり、System of Recordはその作業場であると同時に証跡の保管場所でもある。

**Agent統治レイヤーを備えるSaaS**。Agentを安全に動かすための権限・監査・評価・承認の仕組みをプロダクトに組み込んでいる企業。これは従来SaaSの延長というより、**Agent運用基盤SaaS**という新カテゴリである。

**Vertical SaaS**。深い業界知識をプロダクトに埋め込める企業は、Agentが直面する業界固有の例外・規制・用語・暗黙知・承認慣行に対処できる。ここはFDE的な現場知が直接プロダクトに変換される領域だ。

**FDE→プロダクトの還流ができるSaaS**。個別導入で終わらず、毎回の導入から「共通するAgentパターン」「評価セット」「MCPコネクタ」「承認フロー」「例外処理」「業界テンプレート」を抽出できる会社が最も強い。

## 6. では、あなたはどこに行くべきか

この分析を踏まえたキャリア判断の枠組みを提示する。

### 現職に残る価値がある条件

以下に多く当てはまるなら、現職でAI Agent開発を進める価値が大きい。

- 顧客の重要業務データにアクセスできる
- APIやイベントログなど、Agentが行動できる接点を作れる
- 顧客と直接話し、業務観察ができる
- PoCではなく本番ワークフローに入れられる
- セキュリティ、権限、監査、評価に投資する意思がある
- 個別導入で得た知見を標準機能に戻す文化がある
- 課金モデルを席課金から利用量・成果単位へ広げる余地がある
- 経営が「AI機能の追加」ではなく「AIが業務を遂行するプロダクト」への変化を理解している

### FDE寄せを考えるべき条件

逆に、以下に当てはまるなら、FDEやApplied AI Solutions側へ移ることを検討すべきだ。

- AI Agentを単なるチャットUIやFAQ機能として扱っている
- 顧客業務に深く入ることを営業・CS任せにしている
- エンジニアが顧客の現場を見られない
- 個別要件をすべて「やらない」で切り捨てるだけで、再利用可能な抽象化に向かわない
- Agentの評価、監査、権限、失敗時の復旧を軽視している
- 既存SaaSの画面中心UXを守ることが目的化している
- 経営がAIによる価格モデル・提供価値の変化を受け入れていない

### 今後12ヶ月のアクションプラン

**3ヶ月**：現職でFDE的な動きを作る。1つの顧客業務を選び、人間の業務フロー、Agentに任せる範囲、必要なデータ・ツール、失敗時の復旧手順、評価データセット、成功KPIを定義し、PoCではなく「週次で実際に使われるAgent」を作る。

**6ヶ月**：1社専用で終わらせず、2〜3社に横展開する。共通部分（Agentロール、ツールシーケンス、承認パターン、失敗モード）と差分（ポリシー、用語、評価基準）を特定し、Tenant Agent Packの原型を作る。

**12ヶ月**：現職がFDE-プロダクト合成モデルを受け入れるなら残る。そうでなければ、OpenAI/Anthropic/GoogleのFDE組織、Vertical AIスタートアップ、あるいはSIerのAI導入部隊へ移る。

## 7. さいごに——最も避けるべきは「FDEかSaaSか」を選んでしまうこと

SaaSから逃げる必要はない。しかし、**旧来のSaaS開発者のままでいてはいけない**。

今後の勝ち筋はこれだ。SaaSの保守性・再利用性・マルチテナンシーを理解しつつ、FDEのように顧客現場へ入り、AI Agentが実際に成果を出すところまで作る。そして、その個別導入の知見を、Tenant Agent Pack、評価基盤、権限モデル、Control Plane、業界テンプレートとしてプロダクト化する。

Gartnerは「Agentic AIプロジェクトの40%以上が2027年末までに中止される」と予測する。理由はコスト超過、価値不明確、リスク管理不足だ。この予測が示唆するのは、**Agent導入の現場実装力と統治の仕組みを両方持つ人材・組織こそが淘汰の波を生き残る**という構図である。

最も避けるべきは「FDEかSaaSか」を二項対立で選んでしまうことだ。**最も価値があるのは、その境界に立つことである。**
