# SkillOpt: Executive Strategy for Self-Evolving Agent Skills

**arXiv:** [2605.23904](https://arxiv.org/abs/2605.23904) | **Code:** https://aka.ms/SkillOpt

## Core Idea

SkillOpt treats an agent's external **skill document** as a trainable state and uses a separate frontier optimizer model to edit it with deep-learning–style controls. The deployed artifact is a compact, portable `best_skill.md` (~300–2,000 tokens) that works without any model weight changes — zero additional inference cost at deployment.

> "If the recurring object of adaptation is the agent's procedure, the skill document itself should be trainable."

The deep-learning analogy is **operational rather than decorative**: rollout batches ↔ training minibatches, edit budget L_t ↔ learning rate, validation gate ↔ held-out validation, rejected-edit buffer ↔ gradient history, slow update ↔ momentum.

## Problem Setup

- **Skill** `s`: a natural-language policy inserted into agent context (system prompt or tool-use memory).
- **Frozen target model** `M`: executes tasks with skill `s`.
- **Harness** `h`: direct chat, Codex, or Claude Code.
- **Score** `r(s)`: scalar from 0 to 1 per task.
- **Splits**: Train D_tr → Select D_sel → Test

## Optimization Loop (Algorithm 1)

1. **Forward Pass — Rollout Evidence**: Target model runs batch of tasks with current skill. Harness captures full trajectories, tool calls, scores.
2. **Backward Pass — Minibatch Reflection**: Failures/successes partitioned into reflection minibatches. Optimizer proposes structured **add/delete/replace** edits.
   - Failure minibatches → missing or corrective rules
   - Success minibatches → preserve what already works
   - Edits merged hierarchically, failure-prioritized
3. **Bounded Text Updates (Learning Rate L_t)**: Edit budget limits edits per step (default L_t=4, cosine decay, floor 2). "Unbounded rewrites can erase useful rules; bounded updates preserve continuity."
4. **Validation Gate & Rejected-Edit Buffer**: Candidate evaluated on held-out selection split. Accepted only if **strictly better**. Rejected edits stored as negative feedback to avoid repetition.
5. **Epoch-Wise Slow/Meta Update**: At epoch boundaries, compare same tasks under previous vs. current skill. Longitudinal guidance written into protected skill section (`<!-- SLOW_UPDATE_START -->`...`<!-- SLOW_UPDATE_END -->`). Separate **optimizer-side meta skill** summarizes edit patterns (not deployed).
6. **Harness-Agnostic Deployment**: Lightweight adapter injects skill into any harness. Same `best_skill.md` works across direct chat, Codex CLI, Claude Code, embodied environments (ALFWorld).

## Experiments

### Setup
- **6 benchmarks**: SearchQA, SpreadsheetBench, OfficeQA, DocVQA, LiveMathematicianBench, ALFWorld
- **7 target models**: GPT-5.5, GPT-5.4, GPT-5.4-mini, GPT-5.4-nano, GPT-5.2, Qwen3.5-4B, Qwen3.6-35B-A3B
- **3 harnesses**: Direct chat, Codex, Claude Code
- **Baselines**: No skill, Human skill, One-shot LLM skill, Trace2Skill, TextGrad, GEPA, EvoSkill
- **Default hyperparams**: 4 epochs, rollout batch 40, reflection minibatch 8, L_t=4 (cosine), strict validation gate, slow update with 20 samples

### Main Results

| Setting | SkillOpt Advantage |
|--------|-------------------|
| **All 52 (model×bench×harness) cells** | **Best or tied-best** in every cell |
| GPT-5.5 direct chat average | +23.5 pts over no skill; +5.4 pts over best per-cell baseline |
| GPT-5.5 Codex harness average | +24.8 pts over no skill; +14.0 over EvoSkill |
| GPT-5.5 Claude Code harness average | +19.1 pts over no skill; +3.2 over EvoSkill |
| GPT-5.4-nano (smallest model) average | +26.7 pts over no skill |
| Largest single-benchmark gain | SpreadsheetBench GPT-5.5: 41.8 → 80.7 (+38.9) |

> "SkillOpt is the best-or-tied entry on every measured cell of the table, with positive gains over the no-skill baseline throughout."

The improvement is **not specific to frontier-scale targets**. Small models benefit most in relative terms.

### Key Ablations
- **Training evidence matters** for procedural tasks (SpreadsheetBench 47.5→78.0 with full training set)
- **Reflection minibatch size** and **rollout batch size** relatively insensitive
- **Edit budget L_t=4** near optimal across benchmarks (smaller loses exploration, larger risks overfit)
- **Validation gate is critical** — removing it drops performance significantly
- **Rejected-edit buffer** and **slow update** both provide distinct, complementary gains (Figure 3)

## Relationship to GEPA

SkillOpt and GEPA are both text-space skill optimizers, but with fundamentally different optimization philosophies:

| Dimension | GEPA | SkillOpt |
|-----------|------|----------|
| Search method | Evolutionary (Pareto front) | Gradient-style (bounded stepwise) |
| Edit mechanism | Mutations informed by ancestral lessons | Structured add/delete/replace from minibatch reflection |
| Validation | Constraint gates + test suite | Strict held-out validation gate |
| Learning rate | Implicit (population diversity) | Explicit (L_t, cosine schedule) |
| Stability mechanism | Pareto diversity + PR review | Validation gate + rejected buffer + slow update |
| Epoch awareness | None | Epoch-wise slow/meta update |
| Deployment | PR against repo | `best_skill.md` file |

SkillOpt's results are generally stronger (+5.4 pts over best per-cell baseline, which includes GEPA in some cells), but GEPA's Pareto-based approach may be more robust for diverse skill portfolios where a single "best" skill is insufficient.

## Significance

SkillOpt represents a convergence of deep learning optimization discipline with prompt/skill engineering. By treating skills as external trainable states with proper validation, learning rates, and momentum, it achieves consistent and substantial improvements across models, benchmarks, and harnesses while maintaining deployment simplicity (a single ~300–2,000 token file).
