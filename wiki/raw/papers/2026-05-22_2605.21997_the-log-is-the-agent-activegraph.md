# The Log is the Agent: Event-Sourced Reactive Graphs for Auditable, Forkable Agentic Systems

**Authors:** Yohei Nakajima (Untapped Capital, activegraph.ai)
**Paper:** arXiv:2605.21997
**Open source (Apache-2.0):** https://github.com/yoheinakajima/activegraph
**Install:** `pip install activegraph` → `activegraph quickstart`
**Docs:** https://docs.activegraph.ai

---

## Abstract — Core Inversion

ActiveGraph inverts the typical agent architecture: instead of a chat loop with logging bolted on, **the append-only event log is the source of truth**; the working graph is a **deterministic projection** of that log; and behaviours (functions, classes, LLM-backed routines, logic on typed edges) **react to graph changes and emit new events**. No orchestrator coordinates — all coordination happens through the shared graph.

This design gives three properties missing from retrieval/summarisation memory systems:

1. **Deterministic replay** of any run from its log.
2. **Cheap forking** that branches a run at any event without re-executing the shared prefix.
3. **End-to-end lineage** from goal down to the individual model call that produced each artifact.

The architecture extends the BabyAGI lineage and prior graph-memory research, but does **not** claim improved task performance — the contribution is the substrate and its guarantees.

## 1. Introduction — Built on the Log, Not the Model

Conventional agent construction grows by accretion: chat loop → tools → rules/guardrails → logging → compressed memory (summaries, embeddings). In that model the log is a byproduct, not the core. ActiveGraph reverses it: **the log is the agent itself**.

> What if the rules, tool calls, and content were all events in a single append-only log, and the agent's behavior were just reactions that fire when that log grows and write new events back into it?

ActiveGraph is a direct successor to BabyAGI's while-loop over a task list, converted into reactive behaviors over a durable, inspectable, replayable graph.

### Contributions (architectural, not task-performance)

1. **Event-sourced agent model** — graph state is a deterministic fold over an append-only event log; all action is reactive behaviours emitting events.
2. **Determinism contract and replay** — byte-reproducible runs with a content-addressed cache that records model/tool responses, avoiding new calls during replay.
3. **Forking and structural diff** — branch a run at any event and answer "what if I had done X differently" without re-executing the shared prefix.
4. **Worked, reproducible example** (investment diligence) — the entire causal chain from goal to memo is recoverable from the event log.

## 2. Architecture

### Graph is a Projection of the Log

- **Append-only event log** is the source of truth.
- **Graph state** (typed objects + relations) is **never mutated directly**; it is folded from the log via replay.
- Two replays of the same log produce **deterministic identical state**.

### Events

Every event has: `id`, `type`, `payload`, `actor`, optional `caused_by` (event that triggered it), timestamp. Every object records the behaviour that created it and the causing event — full provenance.

Example event types: `pack.loaded`, `goal.created`, `behavior.started`, `object.created`, `relation.created`, `object.patched`, `llm.request`, `llm.response`, `tool.called`, `tool.result`.

### Behaviours

A behaviour is a **reaction** with:
- **Subscription**: event type, optional predicate, and a **graph-shape pattern** (Cypher subset) — e.g., "a claim that addresses an unanswered question."
- **Body**: fires with triggering event, graph view, and context. Can create objects/relations, apply patches, call tools or models. Everything recorded as events.

Four body forms:
1. **Plain function** — synchronous computation
2. **Class** — carries config, stateful across invocations
3. **LLM‑backed routine** — request & response logged as events
4. **Relation‑behaviour** — logic attached to a typed edge (e.g., "when a claim `supports` an unanswered question, trigger review")

### Why a Graph (Not Just a Log)

1. **Subscriptions are graph‑shape patterns**, not just event types — express conditions like "a claim that addresses an unanswered question."
2. **Relation‑behaviours** attach computation to edges between objects, which a flat projection cannot express naturally.
3. **Structural diff** is a diff over graph topology, well-defined because the projection is a graph.

The log gives reproducibility; the graph gives expressible reactivity and comparison.

### No Workflow — But Not No Coordination

There is **no top‑level script**. The diligence example produces 93 objects and 76 relations from 7 behaviour types, all driven by graph-state subscriptions. Coordination is emergent from the shared graph state.

## 3. Deterministic Replay

Replay folds the event log forward with a **content-addressed cache** (`deterministic=True`). During replay:
- Model/tool calls check the cache first.
- Cache hit → return stored response (no new API call, no nondeterminism).
- Cache miss → call the model/tool, store response, continue.

This guarantees byte-identical state reconstruction from any log.

## 4. Forking and Structural Diff

Forking branches a run at any event:
1. Copy the event log up to the fork point.
2. Append alternative events after the fork.
3. Replay each branch independently.

**Structural diff** compares two graph snapshots (e.g., original vs fork) to show what objects, relations, and properties differ — answering "what would have changed if I had done X differently" without re-executing the shared prefix.

This enables: hypothesis testing, strategy comparison, policy evaluation, and branching reasoning paths — critical for long-running agents that explore alternatives.

## 5. Worked Example: Investment Diligence

The `activegraph quickstart` reproduces a full investment diligence workflow on "Northwind Robotics":

- 7 behaviour types: company_planner, question_generator, web_researcher, claim_extractor, evidence_validator, contradiction_detector, memo_writer.
- 93 objects created (companies, questions, claims, documents, evidence items, contradictions, memos).
- 76 relations (addresses, supports, contradicts, derived_from, depends_on).
- Every step traceable: goal → question → search → claim → evidence → contradiction → memo.

The entire causal chain from goal to memo is recoverable by querying the event log or traversing the graph.

## 6. Significance

ActiveGraph does not claim improved task performance over existing agent frameworks. The contribution is architectural: a substrate that provides **deterministic replay, cheap forking, and end-to-end lineage** for agentic systems. These are properties that retrieval-and-summarization memory architectures (vector DBs, embeddings, summaries) fundamentally cannot provide.

The paper formalizes ideas that Nakajima explored in earlier blog posts (the two-part statefulness series) and the BabyAGI 3 three-layer memory architecture, but adds the formal event-sourcing model, content-addressed cache for deterministic replay, Cypher-based graph-shape subscriptions, and the forking/structural-diff mechanism that were not present in the earlier work.
