# Executable Code Actions Elicit Better LLM Agents

**Authors:** Xingyao Wang, Yangyi Chen, Lifan Yuan, Yizhe Zhang, Yunzhu Li, Hao Peng, Heng Ji
**Venue:** ICML 2024
**Links:** [arXiv](https://arxiv.org/abs/2402.01030) | [GitHub](https://github.com/xingyaoww/code-act) | [Dataset](https://huggingface.co/datasets/xingyaoww/code-act) | [Model](https://huggingface.co/xingyaoww/CodeActAgent-Mistral-7b-v0.1)

## Abstract

LLM agents are typically prompted to produce actions by generating JSON or text in a pre-defined format, which is usually limited by constrained action space and restricted flexibility. This work proposes to use **executable Python code** to consolidate LLM agents' actions into a unified action space (CodeAct). Integrated with a Python interpreter, CodeAct can execute code actions and dynamically revise prior actions or emit new actions upon new observations through multi-turn interactions. Extensive analysis of 17 LLMs on API-Bank and a newly curated benchmark M³ToolEval shows that CodeAct outperforms widely used alternatives (**up to 20% higher success rate**).

## CodeAct Framework

### Three Roles
- **Agent**: receives observations from user (NL) or environment (code execution results), emits either Python code action or NL response
- **User**: provides task instructions in natural language
- **Environment**: Python interpreter that executes code and returns results

### Key Advantages

| Advantage | Description |
|-----------|-------------|
| Multi-turn adjustment | Dynamically adjust prior actions based on execution results |
| Leverage existing packages | Use readily available Python packages for expanded action space; enables self-debugging |
| Code pre-training familiarity | LLMs already extensively exposed to code during pre-training |
| Control & data flow | Loops, if-statements, variables enable composing multiple tools in one action |
| Unified action space | All actions are Python code — no need for separate text/JSON/tool-call formats |

### System Prompt Design
The agent receives a system prompt instructing it to use `<execute>` tags for Python code. The environment (Jupyter Notebook kernel) executes the code and returns output.

## Empirical Evidence

### RQ1: Atomic Tool Calls (API-Bank)
Even for simple single-tool-call tasks, CodeAct achieves comparable or better performance. Open-source models benefit more (Llama-2-7b: 28.8% CodeAct vs 11.3% JSON). Closed-source models like GPT-4 may have been fine-tuned for JSON.

### RQ2: Complex Tool-Use (M³ToolEval)
New benchmark of 82 human-curated tasks requiring multiple calls to multiple tools across domains. CodeAct significantly outperforms text/JSON, especially as task complexity increases.

### RQ3: CodeAct Performance Scaling
Performance gains of CodeAct over JSON/text grow as LLM capability increases — stronger models gain more from CodeAct.

## CodeActInstruct Dataset
7,000 multi-turn interaction trajectories using CodeAct format. Used to fine-tune CodeActAgent models.

## CodeActAgent Models
- **CodeActAgent-Mistral-7b-v0.1** (recommended): Based on Mistral-7B, 32k context, available on HuggingFace + Ollama
- **CodeActAgent-Llama-7b**: Based on Llama-2-7B, 4k context
- Both trained on CodeActInstruct + general conversation data
- Excel at out-of-domain agent tasks while maintaining general performance

## Deployment
Three components: (1) LLM serving via vLLM or llama.cpp, (2) Docker-based Jupyter kernel gateway for code execution, (3) interaction interface (Python script or Chat-UI with MongoDB).

## Key Quote
> "CodeAct employs Python code to consolidate all actions for agent-environment interaction. In CodeAct, each emitted action to the environment is a piece of Python code, and the agent will receive outputs of code execution (e.g., results, errors) as observation."
