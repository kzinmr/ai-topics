## Abstract

We study allowing large language models (LLMs) to process arbitrarily long prompts through the lens of inference-time scaling. We propose Recursive Language Models (RLMs), a general inference paradigm that treats long prompts as part of an external environment and allows the LLM to programmatically examine, decompose, and recursively call itself over snippets of the prompt.

## Key Findings (Updated for v3)

- RLMs can successfully process inputs up to **two orders of magnitude beyond model context windows** (10M+ tokens)
- Dramatically outperform vanilla frontier LLMs and common long-context scaffolds across 4 diverse long-context tasks
- Comparable cost to vanilla inference; median RLM run is cheaper than median base model run
- Post-trained model RLM-Qwen3-8B outperforms underlying Qwen3-8B by **28%** on average
- Approaches the quality of vanilla GPT-5 on 3 long-context tasks
- **RLM training exhibits length generalization** (MRCRv2: 64K → 1M)
- **Depth scaling**: Higher recursion depth significantly improves information-dense tasks (OOLONG-Pairs: 58%→76% from depth 1→3)

## Core Concept

RLMs are a new inference-time scaling paradigm where the LLM recursively accesses and manipulates its own context. Unlike traditional RAG or prompt engineering where context is fixed at inference time, RLMs allow the model to:

1. **Read** its own generated intermediate outputs
2. **Write** computed representations back into context
3. **Recursively refine** outputs through self-feedback loops

## Core Mechanism

A recursive language model is a thin wrapper around a LM that can spawn recursive LM calls for intermediate computation. The key insight: instead of feeding the entire prompt at once, the LLM recursively decomposes and processes the context in chunks, then combines results.

This takes a **context-centric view** rather than a **problem-centric view** of input decomposition.

## Problem Addressed: Context Rot

Anthropic defines context rot as "when the number of tokens in the context window increases, the model's ability to accurately recall information from that context decreases." RLMs directly mitigate this by recursively processing context chunks rather than presenting all at once.

## Technical Mechanism

RLMs work by generating **Python-like code** that can be executed to read/write context:

```python
# RLM-generated code (simplified)
context = read_context()           # Access all prior context
new_repr = compute(context)        # Model processes and generates representation
write_context(new_repr)            # Write back for next step
continue_generation()              # Recurse
```

The key difference from standard tool-use: the RLM is **modifying its own processing context**, not calling external APIs.

## Key Observations (from v3)

1. **RLMs scale to 10M+ tokens** — outperform base LMs and existing task-agnostic scaffolds on long context tasks
2. **REPL is necessary for long inputs; recursion for information-dense inputs** — On CodeQA with Qwen3-Coder, depth=0 (no sub-calls) beats all sub-calling variants. On OOLONG-Pairs, higher depth excels.
3. **LM performance degrades with length × complexity; RLM scales better** — Benchmarks S-NIAH (constant), OOLONG (linear), OOLONG-Pairs (quadratic) show RLM advantage grows with complexity
4. **Cost comparable or cheaper** — median RLM run cheaper than median base model run; cost scales with task complexity but stays in same order of magnitude
5. **RLMs enable longer reasoning** — On LongCoT-mini, RLM + decomposition hints achieve 69.5% overall improvement over GPT-5.2 base
6. **Training cross-generalizes + length generalizes** — RLM-Qwen3-8B trained on LongBenchPro improves across all tasks; MRCRv2 training generalizes 64K→1M

## Error Analysis (§5, expanded in v3)

### First Decomposition & Errors
- RLMs defer unbounded-length reasoning chains to sub-LM calls
- **First decomposition attempt is critical** — while RLMs frequently recover from incorrect initial decomposition, it significantly impacts overall performance
- In-context RLM trajectory examples in system prompt improve both overall performance and initial decomposition quality

### Syntax Errors
- From RLM(depth=1) runs in Table 1, bucketed by correct/incorrect rollouts
- Percentage of trajectories with at least one syntax error is reported per bucket
- Syntax errors are a failure mode that training (RLM-native post-training) reduces significantly

### Decomposition Mistakes
- Models commonly attempt to probe context, then decompose task into sub-tasks
- On BrowseComp-Plus: model uses priors to programmatically narrow search space
- On OOLONG-Pairs: output beyond context window by stitching sub-LM calls in REPL is required

## Key Distinction from DSPy

| Dimension | DSPy | RLMs |
|-----------|------|------|
| Optimization timing | Compile-time (requires training data) | Inference-time (zero-shot) |
| Context management | Fixed prompt with demonstrations | Dynamic recursive access |
| Control body | External Teleprompter | LM itself (via code generation) |
| Data requirements | 10-50+ training examples | None (pure inference) |
| Replayability | High (deterministic compiled prompts) | Low (non-deterministic recursion) |
| Best for | Repetitive pipelines (QA, RAG, classification) | Long-form reasoning, exploration |
| Context scale | Limited by prompt size | Scales to 10M+ tokens via recursion |

## Relationship to Other Approaches

| Scaffold | Context Handling | Info Loss | Trainability |
|----------|------------------|-----------|--------------|
| Direct prompt | All-at-once | None | N/A |
| Summarization | Compressed | High | Hard |
| Retrieval agents | Selected chunks | Medium | Medium |
| OpenCode | File offloading | Low | Medium |
| Claude Code | File offloading | Low | Medium |
| **RLM** | **REPL + recursion** | **None** | **High (end-to-end RL)** |

## Training Details (v3)

### RLM-Qwen3-8B (SFT)
- Fine-tuned Qwen3-8B on 1,000 filtered RLM(Qwen3-Coder-480B-A35B) trajectories from LongBenchPro
- Outperforms base Qwen3-8B as RLM across all tasks
- **3× faster and cheaper**: fewer mistakes, better decision making

### MRCRv2 (RLVR)
- Qwen3-4B-Instruct-0527, RLVR on 32K-64K token split with 2 needles
- 150 steps, batch size 128, 4 rollouts per example
- Max output 4096 tokens/turn, max 20 RLM iterations
- **Length generalization**: achieves strong performance on 1M, 8-needle split
- Comparable to Gemini 3.1 Pro (native 1M context frontier model)
