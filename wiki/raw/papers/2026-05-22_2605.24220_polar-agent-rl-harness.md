# Polar: Agentic RL on Any Harness at Scale

## Abstract

Reinforcement learning for language agents increasingly depends on custom harnesses that manage long-running context, multi-turn tool use and multi-agent orchestration. However, porting these harnesses into RL environment interfaces remains difficult and often loses important training signals. Polar bridges this gap: a rollout framework for scalable asynchronous RL over arbitrary agent harnesses. Polar treats the agent harness as a black box — it proxies LLM API calls, records token-level model interactions, and reconstructs token-faithful trajectories for training. Each rollout node efficiently manages runtime prewarming, agent execution, trajectory reconstruction, and evaluation in parallel, exposing asynchronous service endpoints consumed by independent trainers at scale.

> **"Can we train agents with RL without opening the box?"**

## Core Insight

Every LLM-based agent must call a model through a well-defined API. That API boundary provides a universal observation point independent of the harness. Polar places a provider-compatible proxy below the agent that captures prompts, sampled token IDs, log probabilities, and responses — converting them into RL trajectories without any harness code changes.

## Architecture

Polar decouples rollout generation from training via two main components:

### Rollout Server
- Accepts `TaskRequest`, fans out into independent sessions
- Dispatches to gateway nodes, persists results, exposes polling status, accepts callbacks
- Trainer-agnostic: pairs with Slime but works with any async RL trainer

### Gateway Node
- Owns session lifecycle: runtime startup → harness execution → proxy capture → trajectory reconstruction → evaluation → teardown
- **Isolated worker pools** pipeline phases:
  - `INIT` → prepare runtime, install dependencies
  - `READY` → hold initialized runtimes until execution slot free
  - `RUNNING` → execute harness (GPU-bound)
  - `POSTRUN` → reconstruct trajectories, evaluate, callback, teardown
- Evaluator runtimes can be pre-warmed during agent execution
- Shared session deadline enables partial trace recovery on timeout
- Supports **Docker** and **rootless Apptainer** (HPC/Slurm)

### Proxy Capture (Harness Integration)

Instead of modifying agent code, users point the harness to Polar's gateway as LLM API endpoint:

1. **Detect provider API** — Anthropic Messages, OpenAI Chat/Responses, Google generateContent
2. **Normalize** to OpenAI Chat Completions, inject `logprobs=true`
3. **Capture token-level data** — prompt/response token IDs, log probabilities, finish reasons
4. **Return provider-native response** — synthetic streaming for stream-mode harnesses

> "The proxy boundary is intentionally below the agent framework. It does not need to understand how the harness plans, manages tools, or decides when to stop."

A lightweight **harness adapter** supplies configuration and shell commands. Built-in shortcuts: `claude_code`, `codex`, `gemini_cli`, `qwen_code`, `opencode`, `pi`.

## Trajectory Reconstruction

### Token-Faithful Reconstruction
Converts raw model requests into training traces preserving exact token IDs and log probabilities — no retokenization drift.

**Two strategies:**
1. **Per-request reconstruction** — conservative, one trace per model call
2. **Prefix merging** — stitches append-only multi-turn conversations into single contiguous traces, delivering **5.4× faster trainer updates**

Both strategies live in extensible registries for custom trajectory builders.

## Rollout-as-a-Service (Async API)

Polar exposes rollout generation as a service via async HTTP API:
- Trainers submit tasks → receive callbacks with token-faithful trajectories
- Trainer-agnostic contract (demonstrated with Slime for async-first scaling)
- Decoupled: trainer steps only when sufficient evaluated trajectory groups are ready

## Key Contributions

1. **Proxy-based rollout over black-box harnesses** — LLM API payloads as RL interface, no harness modification
2. **Rollout-as-a-service architecture** — asynchronous isolation of task submission, runtime setup, execution, trajectory construction, evaluation, trainer callbacks
3. **Token-faithful trajectory reconstruction** — exact token IDs with prefix merging for 5.4× speedup
4. **Real-world validation** — GRPO training across Codex, Claude Code, Qwen Code, Pi harnesses
