# エージェントスウォームと新しいモデルの経済性

- Source: https://cursor.com/ja/blog/agent-swarm-model-economics
- Published: 2026-07-20
- Fetched: 2026-07-21

---

今年前半、私たちはエージェントを協調させて目標に向かわせるスケーリングの限界を試す実験を行いました。仮説は、これによってタスクの規模と複雑度における新たな段階が開かれる、というものでした。

その中心となったプロジェクトは、[ブラウザをゼロから構築する](https://cursor.com/blog/scaling-agents)長時間稼働のスウォームでした。概念実証としては成功したものの、完成度の高い本ソフトウェアにはほど遠いものでした。

この取り組みは、意図的に実証重視で進めたものでした。私たちは白紙の状態から出発し、[安定して効果的なシステムへと段階的に改善を重ねました](https://cursor.com/blog/self-driving-codebases)。それ以来の目標は、エージェントスウォームを十分に理解し、意図して設計できるようにすることでした。

その進歩を確かめるため、私たちは旧来のスウォームが苦戦したタスクに再び取り組みました。ドキュメントだけを頼りに、RustでSQLiteをゼロから構築することです。

初期結果は有望でした。私たちは旧世代と新世代のスウォームを同じタスク、同じモデル、同じ時間予算で実行し、非公開のSQLテストスイートのどれだけをパスできるかを測定しました。

新しいスウォームは、すべての構成で旧来より良い結果を出しました。Grok 4.5では4時間で80%に到達した一方、旧来のスウォームは迷走し、2時間目に入る前に停止せざるを得ませんでした。

また、どのモデルにどの役割を担わせるかも変えてみました。ある実行では1つのモデルがすべてを担当し、別の実行では最先端モデルが計画を立て、高速で低コストなモデルが作業を担いました。どの構成でも品質はほぼ同じでしたが、コストには非常に大きな差がありました。

## 木と葉

大きなタスクの説明は、自然と木の形をとります。根にある目標が再帰的に分割され、基本的な作業単位へと落とし込まれていくのです。私たちのスウォームには 2 つのロールがあり、どちらもこの木構造に沿った分解を軸に構成されています。

* 最も高性能なモデルを使うプランナーエージェントは、目標を複数の要素に分割し、それぞれを割り振ります。
* 一般に、より高速で低コストなモデルを使うワーカーエージェントは、それらの要素を実行します。

このデザインは、より厳格なオーケストレーションシステムを包摂するものです。問題に固定のトポロジーを押しつけるのではなく、スウォームの形は問題の輪郭に沿って広がり、計算資源とコンテキストはタスクの複雑度に応じてスケールします。

## ツリーがメモリにおいて果たす役割

1つのエージェントがタスク全体を引き受けると、自分だけでツリー全体をたどり、各リーフへ降りていく必要があります。その間ずっと、祖先ノード、現在位置、そしてより大きな目標をコンテキストに保持し続けなければなりません。

私たちは、これが長時間動き続ける単独のエージェントが次第にぶれていく理由を説明していると考えています。目の前の作業に集中すれば全体像を見失い、全体像を保とうとすれば目の前の部分での出来が悪くなってしまうのです。

スウォームでは、プランナーは実装を行わないため、コンテキストが低レベルの詳細で埋まることはありません。また、ワーカーは計画を立てないため、コンテキスト全体を1つの狭い作業に集中できます。

## エージェントのためのバージョン管理システム

Git や Cargo のようなツールは、並行性制御に大まかなロックを用いています。これは開発者1人であれば問題ありませんが、数百のエージェントが同時並行で生み出す作業量には対応できません。

今年前半の browser スウォーム は、Git 上でピーク時に1時間あたり約1,000コミットを記録しました。新しいシステムでは、ピーク時で1秒あたり約1,000コミットに達します。

## 1秒あたり1,000コミットで起こる破綻パターン

### スプリットブレイン設計
互いの存在を認識していない2人のプランナーが、コードベースの別々の箇所で、同じ概念をそれぞれ異なる方法で実装してしまうことがある。

### プランナー間の競合
2人のプランナーが互いの存在を認識しながら、同じファイルをめぐって変更の応酬を繰り返す競合。

### マージ競合
エージェント同士が常に同じファイルでぶつかり、中立な第三者エージェントが介入して解消する仕組み。

### メガファイル
エージェントの作業先として特に集中しやすいファイル。ワーカーエージェントが肥大化したファイルにフラグを立て、外部のエージェントが分解する仕組み。

### 硬直化
エージェントがコアコードに手を入れないことを学習してしまった問題。意図的な破壊的変更を認め、コンパイラが影響を伝播させる仕組み。

## レビューの観点

長時間動作し、かつマルチエージェントで構成されるシステムでは、エラーは蓄積していきます。相関の低い観点を重ね合わせることで高い効果が得られます。レビューに費やす compute は、非常に高い return をもたらします。

## エージェントが環境を形作る

Stigmergy（シスタージー）とは、アリやシロアリのような群生生物が、直接コミュニケーションを取らずに協調する仕組みです。エージェントが環境を形作り、その環境が次のエージェントの行動を形作ります。

Field Guide — エージェント自身が作成する共有コンテキストの実験。完全にエージェントが管理するフォルダで、index.md は開始時にすべてのエージェントに自動で注入されます。

## SQLite の実験

835ページにおよぶ SQLite マニュアル全体を Rust で実装するよう指示。ソースコード、テストスイート、SQLite バイナリ、インターネットアクセスは与えず。

### モデル構成ごとの結果

1. **GPT-5.5 をプランナーとワーカーの両方に使用**
2. **Grok 4.5 をプランナーとワーカーの両方に使用**
3. **Opus 4.8 をプランナー、Composer 2.5 をワーカーに使用**
4. **Fable 5 をプランナー、Composer 2.5 をワーカーに使用**

新しいハーネスは、すべての構成で旧ハーネスを上回りました。Fable 5 のハイブリッドは、最初の1時間でスイートの約3分の2をクリア。最終的にすべての設定が100%をクリア。

旧ハーネス vs 新ハーネス:
- コミット数: Grok 4.5旧実行は2時間で68,000件（新実行の約70倍）
- マージ競合: 旧実行は70,000件超、新実行は4時間で1,000件未満
- crates数: 旧実行は54個（SQLパッケージ3つ含む）、新実行は9個で安定
- コード行数: Fable 5旧スウォーム64,305行 → 新スウォーム9,908行

## モデルの経済性

- ワーカーが少なくとも69%のトークンを占め、多くのケースでは90%を超える
- プランナーのトークンは単価が高いため、費用内訳はトークン内訳と異なる
- Opus 4.8 + Composer 2.5: プランナー（Opus）がトーンのごく一部だがコストの約3分の2を占める
- GPT-5.5単独実行: ワーカーだけで411ドル

## プロンプトとしての仕様

スウォームでは、仕事の単位が仕様になります。スウォームはコンパイラに似たもの — プランナーは目標をタスクツリーとして解析し、段階的に実行可能な作業へと落とし込みます。
