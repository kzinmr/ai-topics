Learning to talk to an agent (the hard way)
My Holy Sh*t moment with agentic LLMs was when I started working with the vibe coding tool Lovable in the Spring of 2025 (yes, I was late to the party). With no software development background, but some industry experience as a solutions engineer, I was stunned at how easy it was to stand up a web app prototype passion project in the span of a couple of hours. 'Alright,' I thought, 'I get the hype.' This stuff is real, it’s here now, and it’s super powerful (shoutout Lovable, great Hex customer).
It wasn’t all roses, though. Working with the agent came with hiccups,
ahem… bug fix death loops
. And like with any new tech, I found myself mired in 30+ tabs of docs and forum posts and LLM threads on how best to work with the vibe coding agent. My natural inclination for using prompts like “just fix the damn button, don’t change anything else!” was insufficient and ineffective, and I felt like a noob.
One blog post, fervidly named
The Prompting Bible
,
blessed be thy prompt
, cleared a lot of the fog and annoyance I had with these death loops by simply giving me frameworks and examples of prompts I could use to debug. I know it sounds silly — using copy-pastable prompts with a non-deterministic model that are generalized enough to work with any project — but more than anything, the examples and best practices helped me build a mental model for working with an agent and led me to treating it more like an expert consultant and less like a code jockey. I suggest you read the Bible, but here are the quick hits that were helpful for me:
Build a broad plan with the agent; don’t sweat the small stuff initially.
Frequently ask for feedback, consult the agent on approaches, industry standards, and best practices.
Feed the agent’s words back to it — i.e., “Meta Prompting.”
When you’re actually building things, start sweating the small stuff. Be detailed and deliberate.
Don’t be afraid to revert changes and explain why. If something is consistently not working, start over.

🧠 Mental models for working with the Notebook Agent
Structured thinking
You don’t need to be overly explicit in your prompts, but there is a sort of garbage-in-garbage-out paradigm with prompting. Some folks suggest starting with clear structures that break down context, tasks, guidelines, and constraints:
Structured Prompt Template
Copy
Context: You are analyzing [dataset description] to help [business objective].
Task: Perform [specific analysis type] to determine [specific question].
Guidelines: Use [techniques/methods/packages], focus on [key metrics/variables].
Constraints: The analysis must [limitations/requirements], avoid [pitfalls/assumptions].
Template for Data Clustering
Copy
Context: You are analyzing customer transaction data to help improve marketing targeting.
Task: Perform k-means clustering to segment customers based on purchasing behavior.
Guidelines: Use Python's scikit-learn, evaluate different numbers of clusters (2-10), and visualize the results with dimensionality reduction.
Constraints: Handle missing values appropriately, normalize features before clustering, and provide business interpretations for each identified segment.
Conversational prompting
Most of the time, though, you’ll prompt the Notebook Agent kind of like how you’d send a DM to a peer. That’s perfectly fine, and is how I prompt most of the time, but notice that the spirit of the structure outlined above is included in the prompt below.
Conversational Data Clustering Prompt
Copy
I'm looking at our customer transaction data and want to improve our marketing targeting. Could you help me segment our customers based on their purchasing behavior? I'd like to use k-means clustering for this. Use scikit-learn to implement the clustering. Try different cluster counts (between 2-10) to find the optimal number. Make sure to handle any missing values in the dataset appropriately, and normalize the features before running the clustering algorithm.
Meta prompting
One of my biggest pieces of advice is to use the agent to craft a concrete plan, then feed that plan back to the agent for execution. With complex multistep analyses, agents are prone to getting lost in all the steps in front of it. Building a plan and executing step-by-step helps the agent focus on the task at hand, and gives you an opportunity to critique/refine the approach as results stream in. And once you have a plan you like, feed that back to the agent.
(Sidebar — we’re actively thinking about how to make your plans more visible to you, the user, and reduce scrolling in the thread with the agent… Stay tuned!)
Building the plan
Copy
I need to analyze customer churn for our subscription service. What data variables should I include, and what analytical approach would be most appropriate? Can you outline a step-by-step plan for this analysis?
Response from the agent
Copy
1. Exploratory Data Analysis
Churn rate trends over time, by segment, and by cohort
Feature correlation analysis and usage pattern identification
2. Feature Engineering
Create recency/frequency metrics, usage velocity changes
Rolling averages for engagement, days-to-churn calculations
3. Predictive Modeling
Logistic regression for interpretability, random forest for feature importance
Time-to-churn survival analysis for more nuanced insights
4. Actionable Segmentation
Risk scoring (high/medium/low churn probability)
Intervention timing recommendations
Executing the plan
Copy

Scope context, be specific
In a vacuum, the prompt is important, but the team intentionally built the Notebook Agent to be deeply integrated with Hex’s environment. That means you can scope context deliberately, like which tables and dataframes to use or which cells to edit. Scoping context helps the agent focus on the right stuff. When you know what data you need, @ tag the relevant table:
@ tagging data
Copy
Meh: "Analyze the customer data."
Good: "Analyze the @customer_transactions table, focusing on the relationship between 'purchase_frequency' and 'customer_lifetime_value' columns."
Or when you want to make changes to a specific cell, scope it in:
@ tagging cells
Copy
Meh: "Adjust the model so it prioritizes the recent explosive growth we saw in the past 6 months"
Good: "[@Prophet Model Components Analysis] Edit this cell where we define the parameters of the Prophet model so it prioritizes the recent explosive growth we saw in the past 6 months"
Specify analysis methods
If you know the exact approach you want to take in an analysis, clearly state the analytical techniques, models, or packages you want to use. Remember, you don’t need to know everything. If you want suggestions, just ask the model for its advice first, then narrow down the plan.
Example
Copy
Meh: "Predict customer churn."
Good: "Build a random forest classifier to predict customer churn, using feature importance to identify the top factors contributing to churn."
Provide context for business impact
Analysis for analysis's sake is a fool’s errand. The agent can help you reason about actionable next steps that your stakeholders and decision makers really want to know.
Example
Copy
Meh: "Analyze marketing channel performance."
Good: "Analyze marketing channel ROI to determine which channels to increase investment in for our Q4 campaign planning. Consider both acquisition cost and customer lifetime value by channel."
Business rules
Many businesses have structured processes and rules for how to do data work. Think things like: coding conventions, color palettes, analytical techniques, data structures, compliance requirements, etc. Your Hex Workspace has a rules file that applies to every agent interaction in the Workspace that essentially acts as injected context to the agent. Below is an example of Hex’s internal rules file (with some private stuff stripped out).
Example
Copy
# AwesomeCo Analytics Workspace Rules
## Critical Data Governance
- PII Handling: Mask email domains in all outputs (show user@****.com)
- EU Data: Filter WHERE consent_status = 'granted' for GDPR compliance
- Financial Access: Only use finance.* tables for board reports
- Test Accounts: Always exclude WHERE is_internal = false AND email NOT LIKE '%@dataco.com'
## Source of Truth Tables
- Customers: warehouse.customers_dim (updates 2am UTC daily)
- Subscriptions: billing.active_subscriptions_v2 (v1 deprecated Nov 2024)
- Usage Events: events.user_actions_cleaned (raw has duplicates)
- Revenue: finance.recognized_revenue (not billing.charges)
- Marketing: marketing.campaigns_attributed (30-day window)
## Business Definitions
- MRR: Sum of monthly_amount WHERE status IN ('active', 'paused')
- Active User: 3+ sessions in rolling 30 days, exclude internal
- Churn Date: First day of month after last payment
- LTV: Revenue until churn / (1 - gross_margin), GM = 0.75
- Qualified Lead: Score > 50 AND company_size > 10
## Calculations & Logic
- Fiscal Year: Feb 1 - Jan 31 (Q1 starts February)
- Cohorts: By signup_week, weeks start Monday
- Attribution: First-touch for acquisition, multi-touch for expansion
- Exchange Rates: Use finance.exchange_rates for month-end conversion
- Growth Rates: Month-over-month using compound formula, not simple
## Required Analysis Patterns
- Time Comparisons: Always show YoY and MoM together
- Segments: Break down by plan_type (Starter/Pro/Enterprise)
- Statistical Significance: Min 100 samples for A/B tests
- Forecasting: Use 13-week rolling average for baseline
- Anomaly Detection: Flag if metric deviates >2 std from 30-day mean
## Data Quality Warnings
- orders table: 2% duplicate rows, use orders_deduped
- user_attributes: NULL company_size for 30% of records pre-2024
- revenue_forecast: Only reliable 30 days out
- marketing_spend: Delayed by 3 days, note in analyses
## Stakeholder Preferences
- Executive Reports: Focus on trends not absolute numbers
- Sales Team: Include pipeline_stage breakdown
- Product Team: Segment by feature_flag exposure
- Finance: Show both GAAP and billing revenue
## Industry Context
- Benchmarks: Target 125% NDR (industry avg: 110%)
- Seasonality: Q4 -20% new sales, Q1 +35% (budget cycles)
- Competition: We're 2x pricing of alternatives, emphasize ROI
- Market Position: #3 by revenue, #1 by NPS in our segment

Data discovery
The Notebook Agent can help you explore your data warehouse and semantic models without needing to remember exact schemas or table names. This is useful when you’re working across unfamiliar domains or want to find the best source of truth for a metric.
How to use it
Ask the Agent to look up semantic objects, tables, schemas, and column definitions that might be relevant to your question. You can also have it suggest which tables to join, or flag potential duplicates in the warehouse.
Sample prompt
Copy
Help me discover data in the warehouse related to churn. List relevant semantic models, schemas, and tables. Summarize what metrics and columns they contain, and recommend the best starting points for analysis.
knowledge
💡pro tip!
The better documented your metadata is, the better the agent will perform at data discovery (and all other tasks). Add metadata manually in the [data browser] or in dbt/snowflake.
Notebook cleanup & dependency mapping
When working in longer notebooks, it’s easy to end up with unused or duplicative cells. The Notebook Agent can help streamline cleanup by analyzing dependencies and surfacing which cells are critical versus safe to remove. This gives you confidence before deleting anything and helps keep notebooks lean.
Editor’s note: I’m rooting for a feature that natively renders Hex’s Graph DAG to do this… If you like that too, let us know and make our dreams reality.
How to use it
Ask the Agent to scan your notebook, identify irrelevant or unused cells, and generate a quick dependency graph. This shows which cells feed downstream outputs, which are redundant, and which don’t connect anywhere. Use the results to decide what to delete or consolidate.
Sample prompt
Copy
Analyze this notebook for cleanup.
1. Scan all cells and flag any that appear unused, irrelevant, or duplicative.
2. Map the dependency graph of cells (which depend on which).
3. Highlight orphaned cells (not used anywhere downstream), duplicative calculations, or intermediate steps that don’t feed final outputs.
4. Give me a concise summary of which cells are safe to delete, which are candidates for consolidation, and which are critical to keep.
Analyzing teammates work
Sometimes you get a project shared with you by a teammate, and you have a vague idea of what they’ve built, but the project contains hundreds of cells and you just want to get a quick summary of what you’re looking at. Ask the Agent to build you a summary.
Sample prompt
This project was just shared with me by a teammate. Can you summarize the approaches they took in this analysis and any insights gleaned from the results?
Copy
This project was just shared with me by a teammate. Can you summarize the approaches they took in this analysis and any insights gleaned from the results?
Cross-project prompt chaining (context seeding)
When starting a follow-on or related analysis, it’s useful for the Agent to begin with context from a prior notebook. Instead of re-explaining everything, you can have the agent generate a portable prompt that you paste into your new notebook. This “seeds” the agent with definitions, assumptions, and prior steps so it can collaborate with you on what you’re working on now.
To do this, run a prompt in the source notebook asking the agent to produce a portable context block. Then paste that into a new notebook to give the agent the background it needs for your next analysis.
Prompt A (run in the source notebook)
Copy
Analyze this notebook and generate a portable context prompt I can paste into a new notebook. The prompt should:
- Restate the core business question in plain English
- Define each key metrics, dataframes, and calculations precisely (filters, windows, denominators)
- Summarize the data sources (tables, views) with example names
- Note important assumptions (freshness, grain, joins)
- Highlight the main outputs and conclusions so far
- End with “Context Notes” about what might be adapted or extended
Copy
Output only the portable prompt in a single code block.
Prompt B (run in the new notebook)
Copy
Seed context:
[Portable prompt from Prompt A goes here]
Copy
Use this context as background for my current analysis.
Work with me on follow-up or related questions, and suggest how I can
extend or adapt the prior work as needed.
