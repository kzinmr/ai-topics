# OpenAI Symphony: Codex オーケストレーションのオープンソース仕様

## 概要
OpenAI は2026年4月27日、**Symphony** と呼ばれる Codex オーケストレーションのオープンソース仕様を発表した。Symphony は、Linear などのプロジェクト管理ボードをエージェントのコントロールプレーンに変えるオーケストレーターであり、Issue トラッカーを always-on エージェントシステムとして機能させる。

## 背景: インタラクティブ・コーディングエージェントの限界
OpenAI 内部の生産性向上ツール開発において、「人間が書いたコードを一切含まないリポジトリ」を構築する実験を行った。すべてのコードを Codex に生成させ、自動テストとガードレールを徹底した結果は成功だったが、次のボトルネックが浮上した：**コンテキストスイッチング**である。

エンジニアが同時に管理できる Codex セッションは 3〜5 個が限界であり、それを超えると生産性が低下する。エージェントは高速だが、人間の注意力がシステム全体のボトルネックになっていた。

## Symphony のアプローチ
### Issue トラッカーを状態機械として使う
Symphony は、チケットのステータスを状態機械として使用し、Linear をコントロールプレーンにする。開いているタスクごとに専用のエージェントワークスペースがマッピングされ、Symphony がタスクボードを継続的に監視して、アクティブなタスクにエージェントが割り当てられるまでループを回す。

- エージェントがクラッシュまたはストールした場合、Symphony が再起動する
- 新しい作業が出現した場合、Symphony が検知してワークを整理する
- 作業はセッションや PR からデカップルされる。1 つの Issue が複数リポジトリにまたがる PR を生成することも、純粋な調査・分析でコードに触れないこともある

### DAG ベースのタスク分解
エージェントは実装中に、現在のタスクの範囲外の改善点を発見することがある。Symphony はこれを新しいサブタスクとしてファイルし、依存関係グラフ（DAG）を構築する。エージェントはブロックされていないタスクから並列に実行を開始するため、作業は自然かつ最適な順序で展開される。

例: React のアップグレードを Vite への移行に依存させるマークをすると、エージェントは Vite 移行完了後にのみ React アップグレードを開始する。

## 成果
- 一部のチームで ** landed Pull Request が 500% 増加**
- コンテキストスイッチングの大幅な削減
- 人間のエンジニアは「 micromanaging 」から「レビューと承認」に集中できるようになった

## 技術的詳細
- オープンソース仕様として公開: https://github.com/openai/symphony
- Linear, Jira などのIssue トラッカーと統合
- エージェントのクラッシュ/ストール検知と自動再起動
- タスク間の依存関係解決と DAG 実行
