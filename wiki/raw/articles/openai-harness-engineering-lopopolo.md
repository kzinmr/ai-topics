# Harness Engineering: Leveraging Codex in an Agent-First World
**Source:** OpenAI Engineering Blog | **Date:** February 11, 2026 | **Author:** Ryan Lopopolo (Member of Technical Staff)
**URL:** https://openai.com/index/harness-engineering/

---

## Core Experiment
Over five months (starting late August 2025), OpenAI team built and shipped an internal beta product with **zero lines of manually-written code**. Everything вЂ” application logic, infrastructure, tooling, documentation, internal utilities вЂ” was generated by Codex agents.

**Key metrics:**
- ~1M lines of code produced by agents
- ~1,500 PRs opened and merged
- Team grew from 3 to 7 engineers
- Built in ~1/10th the time of manual development
- Throughput: ~3.5 PRs/engineer/day
- Product has internal daily users and external alpha testers

## Redefining the Engineer's Role
> "The primary job of our engineering team became enabling the agents to do useful work."

Early bottleneck was **underspecified environments** вЂ” agents lacked tools, abstractions, and internal structure. The fix was never "try harder" but rather: *"What capability is missing, and how do we make it both legible and enforceable for the agent?"*

### The Ralph Wiggum Loop
Agents self-review locally, request cloud/local agent reviews, and iterate until all reviewers pass. Human PR review is optional; agent-to-agent review dominates.

### End-to-End Autonomy
Given a single prompt, agents can:
1. Validate codebase state
2. Reproduce bug & record failure video
3. Implement fix & validate via app driving
4. Record resolution video
5. Open PR в†’ respond to agent/human feedback
6. Remediate build failures
7. Escalate only if judgment required в†’ merge

## AGENTS.md as Table of Contents
The monolithic AGENTS.md approach failed (context crowding, stale rules). The successful pattern:
- **AGENTS.md (~100 lines):** Acts as a map/table of contents
- **docs/ directory:** Structured system of record
- **Progressive disclosure:** Agents start with small stable map, taught where to look next

## Agent-First Legibility
Codebase optimized for agent reasoning first. "Anything not in-repo/context effectively doesn't exist."

- **Tech stack:** Favor "boring" tech (stable APIs, composability, training set representation)
- **Custom helpers over opaque libs:** Sometimes cheaper to reimplement than use opaque upstream libraries
- **Rigid layering:** Types в†’ Config в†’ Repo в†’ Service в†’ Runtime в†’ UI (fixed dependency directions enforced mechanically)
- **Custom linters:** Enforce structured logging, naming conventions, file size limits, reliability requirements

## Throughput Changes Merge Philosophy
> "In a system where agent throughput far exceeds human attention capacity, waiting is more expensive than a bad merge."

- Minimal blocking gates, short-lived PRs
- Test flakes fixed via follow-up runs, not indefinite blocks
- Corrections are cheap; waiting is expensive

## Continuous Refactoring & Garbage Collection
Agents replicate existing patterns (even suboptimal ones). Manual cleanup (20% of week) didn't scale.

**Solution:** Encode "golden principles" + run background Codex cleanup tasks.
- Prefer shared utility packages over hand-rolled helpers
- Validate boundaries/typed SDKs (no "YOLO" probing)
- Background agents scan for deviations, update quality grades, open targeted PRs (<1 min review, automerge)

> "Technical debt is like a high-interest loan: it's almost always better to pay it down continuously in small increments than to let it compound."

## Open Questions

- Where does human judgment add the most leverage?
- How to encode judgment so it compounds over time?
