# Will Brown @willccbb — RL-Harness Lifecycle Analysis

**Source tweet:** https://x.com/willccbb/status/2045958417073029546
**Date:** 2026-04-30
**Analyst:** User (Kazuki Inamura)

## Full Analysis

### 一言でいうと
強いAIエージェントは、モデル単体のRLでも、外付けハーネス単体でもなく、ハーネスが「訓練環境」になり、その環境でRLされることで次のハーネスが可能になる、という循環で進化する。

Will Brown は Prime Intellect で agentic reinforcement learning の研究・基盤、特に verifiers や Environments Hub に関わっている人物。この投稿は単なる感想ではなく「LLMエージェントを訓練する環境をどう作るか」という実務的な観点から読むべき。

Prime Intellect の verifiers では、環境は「タスク入力」「モデル用ハーネス、つまりツール・サンドボックス・文脈管理など」「報酬関数・ルーブリック」を含むものとして説明されている。

### まず用語をほどく

- **RL** は、モデルが環境内で行動し、その結果に対する報酬で学習する強化学習。数学問題なら正解/不正解、コードならテスト通過、ブラウザ操作なら目的達成。エージェントなら、複数ステップのツール使用や意思決定が報酬対象。
- **Harness** は、モデルそのものではなく、モデルを取り囲む実行系。具体的には、プロンプト、ツール呼び出し、ファイルシステム、ブラウザ、サンドボックス、会話履歴、コンテキスト圧縮、メモリ、サブエージェント、権限管理、ログ、評価器など。
- **Env / environment** は、このハーネスを含んだ「訓練・評価用の世界」。モデルに何を見せ、どんな行動を許し、何を成功とみなし、どう報酬を返すかを定義したもの。

だから冒頭の "it's not RL vs harnesses, it's the RL-harness lifecycle" は、「モデルをRLで強くする派」と「外側のエージェント実装を工夫する派」の対立ではない、という意味。実際には、ハーネスがRLの舞台を作り、RLされたモデルがそのハーネスをよりうまく使えるようになり、さらに高度なハーネスを可能にする。この循環が本質。

### "openclaw envs" の含意

ここでの "envs" は、OSの環境変数ではなく、ほぼ確実に **RL用の環境**。つまり「OpenClaw のような実世界寄りのエージェント基盤を模した訓練環境・評価環境」。

OpenClaw は、単なるチャットUIではなく、エージェントがメッセージを受け取り、文脈を組み立て、モデル推論を行い、ツールを実行し、応答を返し、状態を永続化するループを持っている。公式ドキュメントでも、agentic loop は "intake → context assembly → model inference → tool execution → streaming replies → persistence" と説明されている。また OpenClaw は Gateway が WhatsApp、Telegram、Slack、Discord、Signal、iMessage、WebChat などのメッセージング面を所有する構成で、モデル、ツール、セッション、チャンネルを束ねる実行基盤になっている。

つまり "openclaw envs" とは、たとえば次のようなものを指す:
- あるAIに、Gmailを整理させる。GitHub issueを triage させる。Slackで返答させる。ファイルを読ませる。ブラウザを操作させる。失敗したらログを見て修正させる。必要ならメモリに保存させる。最後に「目的を達成したか」「危険な操作をしなかったか」「余計なコストを使わなかったか」を評価する。

こういう一連の訓練用シミュレーションやサンドボックスが "OpenClaw envs"。投稿者は「フロンティアラボは今、こういう実世界エージェント用の環境構築にいくら使っていると思う？」と聞いている。含意は「かなり使っているはず」。なぜなら、次のAIエージェント性能の差は、単にモデルサイズではなく、どれだけ現実的なエージェント環境でロールアウトを集め、報酬を与え、訓練できるかで決まるから。

### "harness paradigms can only evolve half a model generation at a time"

ここが一番含蓄のある部分。ハーネスのアイデアは、現在のモデルでまったく動かないと、誰も真剣に訓練対象にしてくれない。ログも取れない。成功例も出ない。報酬設計もできない。投資判断もされない。一方で、現在のモデルで完璧に動くなら、それはもう単なるプロダクト機能であり、競争優位としては薄い。みんなが同じように実装できる。一番価値があるのは、**今のモデルでは半分しか動かないが、RLで訓練したら化けるハーネス**。

だから "half a model generation at a time"。モデル世代 N が、ハーネス H をぎりぎり使える。そこでロールアウトを集め、RLで訓練する。するとモデル N+0.5 あるいは N+1 はハーネス H をかなり自然に使えるようになる。そうすると、さらに少し難しいハーネス H+1 が可能になる。このように、モデルとハーネスは交互に足場を作りながら進む。

これは、AIエージェント開発における「足場化 → 訓練 → 内面化 → 次の足場化」というサイクル。

### "the best harness ideas are those which didn't actually work yet"

ここで言っているのは、「今すぐプロンプトだけで安定動作するもの」よりも、「現状ではぎこちないが、訓練されたモデルなら大きな能力差になるもの」が重要だ、ということ。ハーネスは単なるラッパーではない。モデルに新しい行動空間を与えるもの。ツールを呼べる、ファイルを編集できる、別エージェントに仕事を振れる、長い文脈を圧縮できる、メモリを書ける、再帰的に自分を呼べる。これらはすべて、モデルにとって「次に何するか」の選択肢を増やす。

しかし、行動空間を増やすだけでは性能は上がらない。むしろ失敗も増える。だからRLが必要。「いつツールを使うか」「どの情報を保存するか」「どのサブタスクを別エージェントに投げるか」「圧縮時に何を残すか」「並列に走らせるべきか逐次でよいか」を、試行錯誤と報酬で学ばせる必要がある。

つまり、最高のハーネス案とは、現在のモデルにとっては不安定な足場だが、RLで方策を学ばせれば巨大な能力差になる設計。

### 挙げられている例の意味

#### CoT (Chain of Thought)
中間推論ステップを生成させることで複雑推論を改善するプロンプト技法。元論文は、大規模言語モデルに中間推論ステップを生成させると算術・常識・記号推論で性能が改善することを示している。最初は「外から推論形式を指定するハーネス」だったものが、後の推論モデルでは、より訓練された推論行動として内面化されていく。

#### ReAct
Reasoning と Acting を交互に行う枠組み。考える、ツールを使う、観測を得る、また考える、というループ。ReAct論文は、推論トレースとタスク固有行動をインターリーブすることで、外部知識ベースや環境と相互作用できるようにする、と説明している。これも最初はプロンプト・ハーネスだったが、現在のエージェントRLではほぼ標準的な行動様式になっている。

#### Parallel Tools
複数のツール呼び出しを同時に出す能力。OpenAIのツール呼び出しドキュメントでも、モデルが一回のターンで複数関数を呼ぶことがあり、parallel_tool_callsをfalseにするとそれを抑制できると説明されている。しかし本当に難しいのはAPI仕様ではなく、「どのタスクを並列化すべきか」「結果をどう統合するか」「並列実行でコンテキストやコストを浪費しないか」。これはRL向きの判断問題。

#### Claude Code
モデルをコードベース、ターミナル、ファイル編集、テスト、権限、コンテキスト管理の中で動かすハーネス。Claude Codeには /compact のように会話を要約して文脈を解放する機能や、サブエージェントを管理する機能がある。つまり Claude Code は「モデル + 開発環境 + ツール + コンテキスト管理 + ワークフロー」の総体であり、まさにRL-harness lifecycleの題材。

#### Compaction
長くなった会話や作業履歴を要約し、重要な状態だけを残す仕組み。Claude Codeの/compactは「ここまでの会話を要約してコンテキストを解放する」と説明されている。ただし、何を残し、何を捨てるかは非常に難しい。重要な制約、未解決のバグ、ユーザーの意図、作業中の仮説を落とすと、後の行動が壊れる。これも「RLで学ばせる価値があるハーネス判断」。

#### Subagents
別コンテキストで動く専門エージェントに作業を委譲する仕組み。Claude Codeの公式ドキュメントは、サブエージェントを「特定タスクを扱う専門AIアシスタント」とし、メイン会話を大量の検索結果・ログ・ファイル内容で汚さず、別コンテキストで処理して要約だけ返すものと説明している。しかし、どのタスクを誰に投げるか、どれだけ背景を渡すか、返ってきた結果を信頼するか、複数結果をどう統合するかは未解決。ここもRL対象。

#### RLMs (Recursive Language Models)
長いプロンプトを外部環境の一部として扱い、モデルがそれをプログラム的に調べ、分解し、自分自身を再帰的に呼び出す一般的推論パラダイムを提案。これは「長文脈を全部コンテキストに詰める」のではなく、「文脈を外部環境として操作する」発想。これもハーネスとしては不安定でも、訓練されれば非常に強い。

このリスト全体が示しているのは、最初は外付けの工夫だったものが、十分に使われ、評価され、訓練されると、モデルの自然な能力に近づくという歴史。

### "bolt-on memory" への悲観

ここでの "bolt-on memory" は、「モデルに後からベクトルDB、Markdownメモリ、会話履歴検索、ユーザープロファイル保存などをくっつける」タイプのメモリ。

OpenClawのメモリも、モデル内部に隠れた状態があるのではなく、エージェントのワークスペースにMarkdownファイルとして保存されたものを読む、という形で説明されている。公式ドキュメントは「モデルが覚えているのはディスクに保存されたものだけで、隠れた状態はない」と明記している。また OpenClawには、コンパクション前に重要な文脈をメモリファイルへ保存する自動 memory flush もある。

しかし投稿者が悲観しているのは、メモリ一般ではなく、**訓練ループと結びついていない後付けメモリ**。RLでうまく訓練するには、きれいなロールアウトループが必要。

- コード修正なら、モデルがファイルを編集し、テストを走らせ、通れば報酬。
- 数学なら答えが合えば報酬。
- ブラウザタスクならフォーム送信や予約完了で報酬。

これらは比較的「何が良かったか」を評価しやすい。メモリは違う。今日保存したメモリが、3週間後に役立つかもしれない。あるいは、古くなって害になるかもしれない。ユーザーの好みは変わる。どの記憶が成功に寄与したか分かりにくい。誤った記憶を書き込むと将来のセッションを汚染する。プライバシーや安全性の問題もある。検索で引いた記憶が、本当に必要だったのか、たまたま文脈を汚しただけなのかも評価しづらい。

だから "not a clean enough rollout loop"。つまり「メモリを書いた → 後で正しく使った → 報酬が上がった」という因果を、大規模に、安定して、低ノイズで測るのが難しい。

この観点では、単に「長期メモリを付けました」「ベクトル検索できます」「ユーザー嗜好を保存します」では足りない。モデルが、いつ記憶し、何を忘れ、何を検索し、どの記憶を信用し、矛盾をどう扱うかを学ぶには、メモリ自体がRL可能な環境に組み込まれている必要がある。

### この投稿の深い含意

この投稿は、AIエージェント競争の主戦場が「モデル単体の賢さ」から「環境・ハーネス・報酬設計」へ移っている、という見立て。

今後の差分は、単に「GPT-Xが強い」「Claude-Yが強い」ではなく、次のようなものになる:
1. どのラボが、現実に近いエージェント環境を大量に持っているか。
2. どのラボが、ツール使用・コード編集・ブラウザ操作・メモリ・サブエージェント・コンパクションを、報酬付きロールアウトとして回せるか。
3. どのラボが、今は半分しか動かないハーネスを、次世代モデルの訓練対象にできるか。
4. どのプロダクトが、単なるUIではなく、モデルを鍛えるための環境そのものになっているか。

この意味で、Claude Code や OpenClaw のようなものは単なるアプリではない。未来のモデルを訓練するための行動空間のプロトタイプ。現在のモデルがそれを不器用に使っているログは、次世代モデルにとっての訓練データになる。

最終的な読み:
1. 強いエージェントは、まずハーネスとして発明される。
2. 最初はぎこちなく、半分しか動かない。
3. しかし、その半分動く部分がロールアウトを生み、報酬設計を可能にし、RLの対象になる。
4. RLされたモデルは、そのハーネスを自然に使えるようになる。
5. すると、さらに野心的なハーネスが可能になる。

この循環こそが、エージェント進化の実体である。

そして、後付けメモリに悲観的なのは、この循環にまだきれいに乗っていないから。メモリが本当に強くなるには、「保存・検索・更新・忘却」が、プロダクト機能ではなく、訓練可能な行動として設計される必要がある。
