# Thariq Shihipar (@trq212)

**Thariq Shihipar** is a Member of Technical Staff at [[entities/anthropic|Anthropic]], working on [[entities/claude-code|Claude Code]]. He is best known for his advocacy of **HTML as the superior output format for AI agent-generated content**, articulated in his May 2026 blog post "Using Claude Code: The unreasonable effectiveness of HTML." His work sits at the intersection of agent communication design, developer tooling, and the practical craft of building with AI.

See [[entities/thariq-shihipar]] for his full profile including his work on the Claude Agent SDK, Skills system, interpretability research, and entrepreneurial background.

## Overview

Shihipar's core contribution in May 2026 is a rethinking of the **agent-to-human communication surface**. While Markdown became the de facto standard for AI-generated output during the early LLM era, Shihipar argues that it creates a reading bottleneck as agents produce increasingly complex, long-form content. His thesis: **HTML is the right medium for the agent output surface**, and the transition from Markdown to HTML represents the next evolution in agent UX design.

> *"Markdown has become an increasingly restrictive format. I find it difficult to read a Markdown file of more than a hundred lines; I want to use Claude to generate richer visualizations, color and diagrams; and I want to be able to share these outputs more easily."*

## Core Ideas: The HTML-First Philosophy

### 5 Reasons HTML Beats Markdown for Agent Output

Shihipar presents five arguments for preferring HTML over Markdown as the default agent output format:

#### 1. Information Density
HTML can natively represent a far broader set of information than Markdown: tables, CSS-styled design data, SVG illustrations and diagrams, JavaScript interactions, code snippets with `<script>` tags, spatial data via absolute positioning / canvas, and embedded images with `<img>` tags.

> *"In my opinion, there is almost no set of information that Claude can read that you cannot efficiently represent with HTML."*

#### 2. Visual Clarity and Ease of Reading
As Claude produces longer specs and plans, large Markdown files become unreadable. HTML documents can be organized with tabs, illustrations, links, and responsive design, making navigation and comprehension effortless. The solution isn't reading less — it's making the work legible enough that you actually want to read it.

#### 3. Ease of Sharing
Markdown files require attachments; most browsers don't render them natively. HTML files can be shared via a simple link — colleagues can open them anywhere, on any device.

#### 4. Two-Way Interactions
HTML allows interactive elements (sliders, knobs) to tweak designs or algorithm parameters directly in the document. Users can add "copy" buttons to export parameters as a prompt to paste back into Claude Code. This creates a **feedback loop between human and agent** that static Markdown cannot provide.

#### 5. Data Ingestion
Claude Code can pull in rich context — file system, MCP connections (Slack, Linear), web via Claude in Chrome, git history — and render it all into comprehensive HTML reports. The richness of the input context maps naturally to HTML's richer output capabilities.

### The Output Surface as a First-Class Design Problem

Shihipar's broader argument is that **the output surface of an agent is a first-class design problem** — not an afterthought. The standard of "agent dumps Markdown → human reads wall of text" is unsustainable at scale. HTML enables agents to express spatial information (diffs, call-graphs, module diagrams) in spatial form rather than flattening it into linear text.

### Token Efficiency Fallacy

Shihipar directly challenges the historical preference for Markdown based on token efficiency. With modern large context windows and cheap inference, the token cost difference between HTML and Markdown is negligible compared to the expressive benefits. As he notes on Lenny's Podcast (May 2026): **"~1% of tokens go to production code. The vast majority go to dashboards, custom interfaces, weekly status updates, and tools for understanding what to build."**

### Disposable Micro-Apps

A practical pattern Shihipar advocates: ask Claude to create **custom, throwaway UIs** for editing specific sections of plans — e.g., a gamified interface for editing a data visualization rules table. Result: "Micro software on top of micro software" — the perfect tool for a specific problem, discarded after use.

### Simple Prompts Over Constraints

His typical prompt: *"Create an HTML file with a plan. Help me visualize. Include excerpts, mockups, code, **whatever is needed** to give me maximum context."* The phrase "whatever is needed" signals trust. Elaborate system prompts often produce worse results.

## Key Work

### "Using Claude Code: The unreasonable effectiveness of HTML" (May 20, 2026)

Published on the official Anthropic blog, this post lays out the HTML-first thesis with concrete use cases across five domains:

| Domain | HTML Application |
|--------|-----------------|
| **Specs & Planning** | Rich canvas for brainstorming, comparing options, creating mockups, writing implementation plans |
| **Code Review** | Render diffs, annotations, flowcharts, module diagrams with inline margin comments |
| **Design & Prototypes** | HTML is expressive for UI design even if the final surface is React/Swift/etc. |
| **Reporting & Monitoring** | Auto-generated dashboards from git history, Slack threads, incident timelines with color-coded severity |
| **Post-Mortems & Debugging** | Trace dumps, log analysis, timeline views |

The post was endorsed by **Andrej Karpathy**, who positioned it as step 3 in a progression: raw text → markdown → HTML → interactive neural videos/simulations. **Simon Willison** noted it caused him to reconsider his default Markdown preference (formed in the GPT-4 8K context era).

### Companion Site
