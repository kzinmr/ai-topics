## Core Philosophy

### The Cloud Agent Thesis

Dabit's most influential contribution is the distinction between **local agents** (running on a developer's machine in their IDE) and **cloud agents** (persistent, remotely-running agents accessible from any interface):

> *"Cloud agents are a different category from local agents, and the differences compound."*

Key arguments:
- **Accessibility:** Cloud agents work from Slack, CLI, API, Linear — not just IDEs. Non-engineers can tag @Devin in Slack to fix documentation or request features.
- **Persistence:** Agents run asynchronously, working while humans sleep.
- **Organizational leverage:** Local agents make developers faster; cloud agents make engineering orgs more capable.
- **Friction elimination:** The bottleneck isn't writing code — it's the friction of getting a change made. Cloud agents eliminate the need for local dev environments, Git setup, and branch management for simple tasks.

### Engineering for Agents That Never Sleep

Dabit predicts a fundamental inversion in how engineering work gets initiated:

> *"At Cognition, 70% of all Devin sessions are currently kicked off by a person... The other 30% start automatically via API, scheduled sessions, and other automations. That ratio will continue to invert. Within a few months it will likely be 30/70. Within a year, 10/90."*

The core insight: **the prompt box is the bottleneck.** Most engineering work is triggered by machine-readable signals (failing tests, latency spikes, approved specs). Instead of a human reading the signal and typing a prompt, the signal should trigger the agent directly.

### Prerequisites for Agent-First Engineering

Dabit identifies the scaffolding needed for agents to work autonomously:

1. **Comprehensive unit tests** — so agents can verify their own output
2. **Good documentation** — so agents have context without asking
3. **Reproducible dev environments** — so agents can run code without custom setup
4. **Rich system context** — so agents understand architecture, conventions, and service interactions

> *"That scaffolding is the difference between an agent that opens a PR and an agent that closes an incident at 3am. Without it, you have agents that write code. With it, you have agents that ship software."*
