## Core Ideas

### Web Minimalism as Practice, Not Theory

Maurycy's entire website is a **manifesto for the small web**, demonstrated through implementation rather than essay:

- Every post is **handwritten in HTML** — no CMS, no framework, no markdown processor
- The homepage, index pages, and RSS feed are generated by a custom SSG in **~1000 lines of C**
- No JavaScript required for core functionality (theme switcher works without it)
- The site works in **text-only browsers like Lynx** — full functionality with zero CSS
- **Gopher protocol support** — because "people need a thousand lines of C code to read my website. This is unacceptable"
- Tracks both publication date and last modification date, so readers can see how old content is
- No comment section — friction (email me) reduces "knee-jerk spam" and reminds people they're talking to a person
- A **kitten on the homepage** that chases your mouse cursor (if not on a phone) — "it's good to have more weirdness on the internet"

This isn't aesthetic cosplay. It's a functional demonstration that a personal website should be **small, fast, and owned by its creator**.

### The Custom Static Site Generator (Jan 2026)

Maurycy wrote his own SSG in C after evaluating Hugo and finding it wanting:

**Why Hugo failed him:**
- Nearly all SSGs force Markdown — "common, but poorly specified and difficult to parse"
- Template languages are "entire programming languages shoved inside HTML: complete with variables, loops, and conditionals"
- Configuration complexity outweighed the benefit

**His solution:**
- Each article is an HTML file with a metadata block:
  ```
  title Example page:
  date 2021-02-11
  edited 2025-01-21
  tag electronics
  tag radiation
  ---
  ```
- The program copies all files from source to destination; everything except HTML passes through unchanged
- Posts sorted by date: first 5 on homepage + RSS, all posts (except photos) on archive page, 12 tag index pages
- A **catalog page** generated from objects tagged in photos
- **Build time: 43 milliseconds** for 100+ pages, 2000+ assets, ~1 GB total
- **Zero external dependencies** — no configuration, no assets, no DSL
- Switching took less time than "configuring Hugo"

> *"It took only a few hours of work to get mine to the point it could replace Hugo"*

This is the small web philosophy in action: **a few hours of honest work replaces megabytes of configuration**.

### "Why Does C Have the Best File API?" (Feb 2026) — 162 HN Points

A provocative essay arguing that C's approach to file I/O — particularly **memory-mapped files** — is superior to every other language:

```c
int file = open("numbers.u32", O_RDWR | O_CREAT, 0600);
ftruncate(file, len);
uint32_t* numbers = mmap(NULL, len, PROT_READ | PROT_WRITE, MAP_SHARED, file, 0);
// Now access numbers[42] directly — it's just memory
```

Key argument: *"These languages have all these nice features for manipulating data in memory, but nothing for manipulating data on disk."* Most languages give you `read()`/`write()` and a serialization library, then stop. C lets you **treat disk as memory** — mmap works even for files larger than RAM, with the OS handling paging transparently.

The broader point: file manipulation is neglected in modern languages. The filesystem is "the original NoSQL database" but languages rarely give you more than a wrapper around `readdir()`.

### "How Helpful Languages Create Bugs" (Jul 2023)

Maurycy argues that language features designed for **conciseness and automation** (dynamic typing, overloading, implicit conversions, garbage collection) often **obscure actual program behavior**:

> *"The readability of languages like C++, C#, Dart, Python, JavaScript, Kotlin, Scala, etc is an illusion. You might think the computer is doing one thing, when in reality it is doing something completely different."*

Concrete example of operator overloading ambiguity:
```cpp
Vector2f multiply(Vector3f a, Vector3f b) { return a * b; }
```
Is `*` piecewise multiplication or cross product? *"Developers split 50/50 on interpretation, proving the syntax is inherently unclear."*

The essay connects language design to a broader philosophy: **explicit code prioritizes human readability and compiler safety over brevity**. When you must declare variables, specify types, and handle errors explicitly, the compiler catches mistakes before runtime.

### Physics Engine from Scratch (Jun 2023)

A complete tutorial series on building a 2D physics engine in C using **Verlet integration**:

```
xn+1 = 2xn - xn-1 + a·Δt²
```

Key insight: instead of storing velocity, compute it from current and last positions. This makes collision resolution trivial — just update position and the velocity is implicitly correct.

The follow-up, **"From 1 fps to 500 fps"**, demonstrates spatial hashing optimization: divide the world into grid cells so each particle only checks collisions against neighbors. Result: **~100x speedup at 5000 objects** without SIMD, multithreading, or assembly — just an algorithmic improvement.

> *"By simply finding a clever way to do less work, the code becomes hundreds (or with very large systems thousands or tens of thousands) of times faster."*

### The 1000-Line Web Browser

Maurycy built a functional web browser in **~1000 lines of C** to prove a point:

| Lines | Function |
|-------|----------|
| 107 | User interface |
| 127 | URL parsing |
| 99+ | HTTP over SSL (via OpenSSL) |
| 90 | HTTP over plaintext sockets |
| 364 | HTML parsing and typesetting |
| **999** | **Total** |

Features: word-wrapped text, bold/emphasized text, code blocks, auto-blocks cookies and JavaScript. Takes a URL argument, assigns numbers to links, exit with Ctrl-C.

> *"Modern browsers are hugely complex: Chromium has 49 million lines of code... but how much of that is needed if I just want to visit websites instead of running multi-gigabyte Javascript applications?"*

### GopherTree — A Gopher Client in 1000 Lines of C

Maurycy's gopher client is a **statement about protocol design**: gopher is "usually seen as a proto-web" but is actually closer to FTP — no markup, no links, no URLs, just hierarchical files. This rigid structure allows clients to display content creatively.

Disappointingly, most gopher clients use a "limited web" aesthetic. GopherTree renders menus as a **directory tree** (hiding informational selectors by default), has built-in text viewer with pagination and word-wrap, supports UTF-8 and Latin-1, and stores bookmarks as local gopher menu files.
