# Hornet

**Hornet** is a retrieval engine purpose-built for AI agent workloads, founded in 2025 by ex-Vespa engineers led by **Jo Kristian Bergum** (CEO). It addresses the fundamental mismatch between legacy search infrastructure (tuned for human keyword queries) and the new reality of agentic retrieval — where LLM agents issue dozens of structured, multi-term queries per session inside iterative reasoning loops.

## Core Thesis

> "Retrieval infrastructure was built and tuned to match the workloads generated by humans. None of those choices were wrong for the workload they targeted. They just do not describe the workload in this trace."

The founding insight: agents are fundamentally different users of search. Where humans issue 2-word queries expecting ranked snippets, agents issue 10+ term structured queries with operators, re-query based on intermediate results, and need full-document context — not snippets.

## Key Characteristics

- **Model-agnostic**: Open source, no vendor lock-in; use any embedding or ranking model
- **Schema-first APIs**: Predictable, structured inputs/outputs designed for programmatic consumption by agents
- **Verifiable**: APIs expose enough information about the retrieval process for agents to audit and improve their own context
- **Self-hosted**: Runs in VPC or on-premises alongside agent infrastructure and data
- **Multi-workload**: Covers single-agent reasoning loops to multi-agent systems at web-scale

## Architecture Philosophy

### API as a File System

Hornet's API surface is structured to "look and feel like a coding environment" because frontier models are increasingly optimized via reinforcement learning to reason through code execution and file systems. Agents write, edit, and read configuration files — much like scaffolding a Next.js app.

### Levels of Agent Autonomy

| Level | Description |
|-------|-------------|
| **Basic** | Agent constructs valid queries using Hornet's retrieval capabilities (standard RAG) |
| **Sophisticated** | Agent configures retrieval settings based on specific needs, adjusting parameters |
| **Highest** | Agent observes its own retrieval quality over time and iteratively tunes configurations |

### Verifiable Feedback Loop

Configurations are like source files → API validation is like a compiler → Behavioral metrics are like test suites → Deployments are versioned releases that can be verified and safely reverted.

## Agentic Retrieval Workload (Empirical Evidence)

From the BrowseComp-Plus benchmark analysis (May 2026):

| Metric | Human Search | GPT-5 Agent Search |
|--------|-------------|-------------------|
| Median query length | 2 terms | 10 terms |
| p90 query length | 5 terms | 17 terms |
| First-query average | — | 19.1 terms |
| Median searches/session | 1 (single-shot) | 24 calls |
| Phrase quotes usage | rare | 98% of sessions |
| `site:` operator usage | rare | 48% of sessions |
| Year operators | rare | 95% of sessions |

Agents also use advanced operator combinations (OR across year ranges, wildcard subdomain, filetype:, negation) that humans almost never type.

## Why Legacy Infrastructure Fails

- **Inverted indexes** tuned for short queries; BM-WAND pruning degrades past ~7 terms
- **Neural retrievers** trained on MS MARCO/Natural Questions where queries are short and fluent — GPT-5's queries are out-of-distribution
- **Search APIs** expose one string because that's what humans typed; operators were deprecated
- **Benchmarks** score single queries, not iterative 24-call sessions
