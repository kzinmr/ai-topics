# Agentic Manual Testing

コーディングエージェントによる**探索的テスト**。自動テストがパスしても見逃される問題（クラッシュ、UI欠落、エッジケース）を発見するために、エージェントの手動テスト能力を活用する。

## 核心原則

> "The defining characteristic of a coding agent is that it can execute the code that it writes."

> "Never assume that code generated by an LLM works until that code has been executed."

## なぜ自動テストだけでは不十分か

自動テストがパスしても、以下の問題を見逃す可能性がある：
- **クラッシュ**: 実行時エラー
- **UI欠落**: 表示上の問題
- **エッジケース**: テストスイートがカバーしていない入力パターン
- **実装の誤り**: テストは通るが、意図と異なる動作

## テストメカニズム（コードタイプ別）

### Pythonライブラリ
```bash
Try that new function on some edge cases using python -c
```

### 他の言語
```bash
Write code in /tmp to try edge cases of that function and then compile and run
```
→ `/tmp`を使用することで、リポジトリへの意図しないコミットを回避

### JSON/Web API
```bash
Run a dev server and explore that new JSON API using curl
```
→ キーワード **"explore"** がエージェントに複数のパラメータ/パスをテストさせる

## ブラウザ自動化（Web UI）

### 主要ツール
| ツール | 開発元 | 説明 |
|--------|--------|------|
| Playwright | Microsoft | 多言語バインディング |
| agent-browser | Vercel | PlaywrightのCLIラッパー |
| Rodney | Simon Willison | Chrome DevTools Protocolベース |

### なぜ効果的か
> "Coding agents natively understand and leverage dedicated CLIs."

### 歴史的課題の克服
従来の「flaky」なブラウザテストが、エージェント時代には保守しやすくなった：
- エージェントがUI変更に自動追従
- テスト更新のdeveloper frictionが大幅に低減

### Rodney プロンプト例
```bash
Start a dev server and then use `uvx rodney --help` to test the new homepage, look at screenshots to confirm the menu is in the right place
```

#### エンジニアリングの工夫
1. **`uvx rodney --help`**: 初回実行で自動インストール + エージェント最適化ドキュメント提供
2. **エージェントファースト `--help`**: CLIヘルプテキストをLLM消費用に構造化
3. **ビジョン活用**: `"look at screenshots"` でエージェントの視覚検証能力を明示的にトリガー

## Showboatとの連携

[[showboat]] ツールを使用して、エージェントのテスト作業を**検証可能・監査可能な成果物**として記録。

> "Forces agents to 'show their work' by creating verifiable, auditable testing artifacts."

## TDDとの連携

探索的テストでバグを発見 → **Red/Green TDD** で永続的な自動テストカバレッジに追加
