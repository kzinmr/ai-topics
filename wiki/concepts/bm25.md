# BM25

BM25 (Best Matching 25) is a classical **lexical retrieval** algorithm that ranks documents by term frequency and inverse document frequency. For decades it was the workhorse of search engines (Elasticsearch, Lucene) but was largely considered "obsolete" for modern AI applications — until PI-SERINI (May 2026) demonstrated that BM25 + a capable LLM in an agentic loop can **beat dense neural retrievers** on deep research benchmarks while being 3.3x–10x cheaper.

> **Key insight (Hsu et al., 2026):** "BM25 is sufficient for deep research when paired with capable LLMs in an agentic loop, challenging the assumption that dense retrievers are necessary."

## Algorithm

BM25 scores a document $d$ against a query $q$ as:

$$\text{BM25}(q, d) = \sum_{t \in q} \text{IDF}(t) \cdot \frac{f(t,d) \cdot (k_1 + 1)}{f(t,d) + k_1 \cdot (1 - b + b \cdot \frac{|d|}{\text{avgdl}})}$$

Where:
- $f(t,d)$ = term frequency of $t$ in $d$
- $|d|$ = document length
- $\text{avgdl}$ = average document length in corpus
- $k_1$ = term frequency saturation parameter (default: 0.9 in Anserini/Lucene)
- $b$ = document length normalization (default: 0.4 in Anserini/Lucene)
- $\text{IDF}(t)$ = inverse document frequency

## BM25 in the Agentic Era

### PI-SERINI: BM25 + LLM Agent Loop

The PI-SERINI paper (Hsu et al., May 2026) showed that BM25 lexical retrieval, when embedded in an agentic search loop with a capable LLM, achieves **state-of-the-art deep research performance**:

**Architecture:**
- Three explicit tools: `search`, `read_search_results`, `read_document`
- LLM caches rankings of up to **1000 documents** and browses them selectively
- Documents are NOT dumped inline — the agent actively chooses what to read
- **Wall-clock budget** (300 seconds) replaces fixed iteration cap, with a submission steer injected at 70% of timeout

**BM25 Tuning for Agentic Use:**
- Standard Anserini defaults ($k_1=0.9$, $b=0.4$) are tuned for short web documents
- For BrowseComp-Plus (long, noisy documents): **$k_1=25$, $b=1$** — dramatic reparameterization
- Retrieval depth increased to **1000** (vs typical top-10/20)

**Results:**
- gpt-5.5 + PI-SERINI: **83.1% answer accuracy**, **94.7% evidence recall** on BrowseComp-Plus
- Beats all released dense-retriever agents
- **3.3x–10x cost reduction** via prefix caching (82–90% of tokens served from cache)
- The agentic loop itself is cache-friendly because the system prompt and tool definitions are static

**Failure Mode: "Premature Branch Commitment"**
- claude-opus-4.7 keeps probing weak hypotheses; gpt-5.5 backtracks to original clues
- This explains part of the gap between otherwise comparable frontier models
- Implication: the LLM's **search strategy** matters as much as retrieval quality

## BM25 vs Dense Retrieval

| Dimension | BM25 (Lexical) | Dense Retrieval (Embeddings) |
|---|---|---|
| **Mechanism** | Exact term matching + TF-IDF statistics | Semantic similarity in embedding space |
| **Cost** | Near-zero (no GPU, no embeddings) | Embedding computation for queries + documents |
| **Cache friendliness** | Stateless, no model inference needed | Embeddings can be precomputed but still require storage |
| **Strengths** | Exact keyword matches, rare terms, transparency | Semantic understanding, synonym handling, multilingual |
| **Weaknesses** | Vocabulary mismatch, no semantic understanding | Embedding model quality dependent, compute cost |
| **Agentic synergy** | Cache-friendly (82-90% prefix cache hit in PI-SERINI) | Higher per-query latency, less cache benefit |

## Why This Matters

1. **Cost efficiency**: BM25 is essentially free compared to embedding-based retrieval. For agentic loops that make dozens of search calls, this compounds.
2. **Prefix caching**: The static system prompt + BM25 as a deterministic retriever means most tokens are cached — a property dense retrievers don't share.
3. **Transparency**: BM25 rankings are interpretable (you can see which terms matched). This helps LLMs reason about search results.
4. **Re-evaluates assumptions**: The field has been moving toward denser, larger embedding models. PI-SERINI suggests the bottleneck is the **agent's reasoning about search results**, not retrieval quality.
