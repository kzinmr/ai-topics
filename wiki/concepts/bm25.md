# BM25

BM25 (Best Match 25) is the dominant scoring function for [[lexical-search|lexical search]], used in virtually every production search engine (Elasticsearch, Solr, Lucene, OpenSearch). It evolved from TF-IDF with three key refinements: **saturated term frequency**, **logarithmic inverse document frequency**, and **document length normalization**.

> "BM25 ≈ Relevance Given Match" — Doug Turnbull

## Evolution: TF → TF*IDF → BM25

### 1. Simple Term Frequency (TF)

```python
def term_counts(term_freqs, ...):
    return term_freqs  # Raw count of term occurrences
```

**Problem:** A document mentioning "doug" twice scores the same as a document mentioning both "doug" and "complaint" — no notion of term rarity or relevance density.

### 2. TF*IDF

```python
def tf_over_df(term_freqs, doc_freqs, ...):
    return term_freqs / doc_freqs  # Down-weight common terms
```

**Inverse Document Frequency (IDF):** Terms appearing in fewer documents get higher weight. "Complaint" (rare) counts more than "the" (common).

**Problems with TF*IDF:**
- Term frequency should **saturate** — the 19th occurrence isn't 19× more important than the 1st
- No document length normalization — "Skywalker" in a tweet ≠ "Skywalker" in a book

### 3. BM25 — The Three Innovations

**Innovation 1: Saturated Term Frequency**

```
tf → tf / (tf + k1)
```

As `tf` increases, the score approaches 1.0 asymptotically. The first mention of a term provides the strongest signal; subsequent mentions provide diminishing returns.

**Innovation 2: Logarithmic Document Frequency**

```
idf = log(1 + (num_docs - df + 0.5) / (df + 0.5))
```

Logarithmic scaling means rarity matters less after a certain point — a term in 1 document isn't infinitely more important than a term in 10 documents.

**Innovation 3: Document Length Normalization**

```
length_bias = (1 - b + b * dl / avg_dl)
```

Accounts for the fact that a term in a short document is a stronger relevance signal than the same term in a long document.

**Combined BM25 formula:**
```
tf → tf / (tf + k1 × length_bias)
score = Σ tf_saturated × idf_log
```

## BM25 Parameters

| Parameter | Range | Effect | Typical |
|-----------|-------|--------|---------|
| `k1` | 0–∞ | TF saturation speed. Low k1 = saturate quickly. | 1.2–2.0 |
| `b` | 0–1 | Length normalization strength. High b = more length influence. | 0.75 |

- **Low k1** (e.g., 0.5): "First match is enough" — good for short fields like titles
- **High k1** (e.g., 2.0): Term frequency still matters — good for long-form content
- **b=0**: No length normalization (tweets, short messages)
- **b=1**: Full length normalization (books, long articles)

## BM25F — Multi-field BM25

When search spans multiple fields with different characteristics (e.g., "title" vs. "body" vs. "topics"), naively combining per-field BM25 scores breaks because IDF is calculated independently per field. A term common in "body" but rare in "topics" gets artificially inflated IDF.

**BM25F solution — two-stage combination:**

1. **Combined TF:** Saturate term frequencies across all fields together
   ```
   saturated_tf = Σ(per_field_tf) / (k1 + Σ(per_field_tf))
   ```
   Length normalization applied per-field first, then combined saturation.

2. **Combined IDF:** Take the maximum IDF across fields
   ```
   idf_combined = max(field1.idf(term), field2.idf(term), ...)
   ```
   Prevents rare-field artifacts from dominating scores.

```
BM25F = saturated_tf_combined × idf_combined
```

## Scaling Behavior

BM25 is computationally efficient — inverted index lookups are O(log n) or O(1) with hash-based indexes. However, query-length to latency follows a power-law distribution, as documented by [[entities/turbopuffer|turbopuffer]]:

- Short queries (1–3 terms): ~1ms
- Long queries (10+ terms): Latency scales super-linearly due to merge overhead
- This matters for [[ai-agents|agentic search]] where agents may generate many long BM25 queries

## BM25 in the Age of AI Agents

BM25 is experiencing a renaissance driven by agentic search patterns:

### Dumb Retriever + Smart Agent

Doug Turnbull's core thesis: Complexity should move from retrieval to the agent. **BM25 is the ideal "dumb" retriever** — fast, deterministic, interpretable, and tunable. The LLM agent adds semantic understanding on top.

```python
# Agentic search loop (pseudocode)
results = bm25_search(query, index)         # Fast lexical retrieval
filtered = agent.reason_about(results)       # LLM semantic filtering
reranked = agent.rerank(filtered, context)   # LLM relevance assessment
return agent.synthesize(reranked, query)     # Final response generation
```

### BM25 as an Agent Tool

In architectures like [[concepts/agentic-search]], BM25 is exposed as a tool the agent can call:

- Agent formulates precise keyword queries for BM25
- Agent iteratively refines queries based on results
- Agent uses BM25 to verify facts ("grep for truth")
- Agent combines BM25 results with vector search results

### Key Insight

> "RAG isn't a vector search problem. Embeddings became the singular framework for RAG, but it's the wrong lens." — Doug Turnbull

The embedding-centric view of RAG over-indexes on semantic similarity and under-indexes on **precision, speed, and interpretability** — all of which BM25 excels at.

## Comparison: BM25 vs. Vector Search

| Dimension | BM25 | Vector/Embedding Search |
|-----------|------|------------------------|
| Speed | <1ms per query | 5–50ms (ANN) |
| Precision | Exact term match | Approximate semantic match |
| Recall | Limited to vocabulary | Broad semantic coverage |
| Interpretability | Full (which terms matched) | None (black-box similarity) |
| Multilingual | Per-language tokenizer needed | Shared embedding space |
| Tuning | Parameters (k1, b) + tokenizer | Model choice + training data |
| Edge cases | Synonyms, typos, paraphrases | Semantic drift, adversarial |

**Best practice:** Use BM25 for precision retrieval (keyword matching, fact verification) AND vector search for recall (semantic discovery, related content). The agent arbitrates between them.
