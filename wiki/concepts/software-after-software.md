# Software After Software

**Software After Software** is a manifesto published by [[entities/amp|Amp Labs]] in May 2026, articulating a comprehensive vision of how AI is transforming the software industry at a structural level. Authored by the Amp team ([[entities/thorsten-ball|Thorsten Ball]], Beyang Liu, and colleagues), it argues that the assumptions underpinning 40 years of software development — that code is hard, scarce, and human-bound — no longer hold.

## Summary

The manifesto advances 12 theses structured as a logical argument:

1. **Everything is changing** — the transformation is so large that all assumptions about software and business must be rethought
2. **Model improvement is still underestimated** — exponential progress means current scaffolding (spellcheckers, training wheels) will become unnecessary
3. **Most code will be written by agents** — the assistant sidebar is dead; agents run autonomously, continuously
4. **The bottleneck has moved** — writing valid code is trivial; errors of engineering (priorities, sequencing, tradeoffs) remain
5. **40 years of software development is dead** — built on assumptions that no longer hold
6. **Old processes become drag** — planning, prioritization, reviews were designed for slow, costly, human-bound implementation
7. **Software changes form** — from human-facing to agent-facing; from pre-built to just-in-time
8. **Value moves away from code** — data, permissions, distribution, trust, compliance become the moats
9. **Winners reorganize around models** — not fitting AI into old processes, but restructuring around models
10. **Every institution needs a camp at the frontier** — small autonomous teams discovering the new way
11. **The engineer changes** — from code-writing to product thinking, system design, business judgment
12. **Therefore** — build for abundant, continuous, cheap intelligence; go to the frontier

## Key Theses

### 1. Code Is No Longer Scarce

> "Software, as an industry, is built on the assumption that code is scarce. Software, as a profession, is built on the assumption that writing code is hard and error-prone. Both assumptions no longer hold."

This economic reframing is the manifesto's central argument. When code becomes abundant — generated by agents freely and continuously — the entire value chain of the software industry shifts. The scarcity that justified SaaS subscriptions, enterprise licensing, and dedicated engineering teams evaporates. What remains valuable are the non-code assets: data, permissions, distribution channels, trust relationships, compliance positioning, and physical assets.

### 2. Agents Don't Work Like Humans — And Shouldn't

> "An agent forced to work like a human is a wasted agent."

Agents will run without a human in front of the computer, for longer periods, with less supervision. The unit of work becomes the **delegated task**, not lines of code. The assistant sidebar model — where a human prompts an AI and waits for a response — is described as already dead. The future is agents running continuously, handling multiple tasks, making decisions within delegated authority.

This directly connects to [[concepts/agentic-engineering]] and the shift from human-in-the-loop to human-on-the-loop patterns.

### 3. Software for Agents, Not Humans

> "Software, so far, was built for human use. In the future, software will be built mainly for agent use."

This is one of the manifesto's most provocative claims. As agents become the primary consumers and producers of software, the form of software itself changes:
- **Just-in-time over pre-built**: More software built when needed, less stored in advance
- **Blurring use/build**: The line between "using software" and "building software" disappears
- **Agent-native interfaces**: APIs, tool definitions, and headless services become primary; human UIs become secondary

### 4. Reorganize Around Models

> "It will not be enough to fit models into existing systems, org charts, and processes."

The winners of this transformation organize **around the models** — not the other way around. A small team with strong judgment and many agents outruns a large team trying to bolt AI onto pre-transformation processes. This echoes the [[concepts/harness-commoditization|harness commoditization]] thesis that the agent harness itself is no longer the differentiator.

### 5. Camp at the Frontier

> "Every serious institution needs a camp at the frontier."

Large corporations cannot learn the future through committee or sit this transformation out. They need a small, autonomous team at the frontier, built around the models. This team's purpose is not to add AI to the old way of working, but to **discover the new way** and pull the organization toward it. Its output is not only software, but also **people and practices**.

This is a concrete organizational prescription: embed a frontier team close enough to real systems, data, and constraints to be credible, but autonomous enough to break old processes.

### 6. The Engineer Doesn't Disappear — But Changes Fundamentally

Software engineering shifts from code toward:
- **Product thinking** — what should be built, not how to build it
- **System shaping** — architecture, constraints, integration design
- **Tradeoff judgment** — priorities, sequencing, cost/benefit at the business level
- **Business outcomes** — the engineer's value is measured by business impact, not code output

## Economic Implications

### What Loses Value
- Software-as-code-someone-else-wrote (SaaS built on implementation scarcity)
- Workflow-encoding software (agents can generate workflows JIT)
- Vendors whose moat was "customers cannot justify building this themselves"

### What Gains Value
- **Data** — proprietary datasets, real-time feeds, unique signals
- **Permissions & Trust** — identity, authorization, audit trails
- **Distribution** — existing customer relationships, platform access
- **Compliance & Regulation** — navigating legal frameworks agents cannot
- **Physical Assets** — hardware, infrastructure, real-world presence

## Reception & Context

The manifesto fits within a broader discourse about the post-AI future of software engineering that includes:
- Tim O'Reilly's "The End of Programming" — the shift from code-writing to system design
- Andrej Karpathy's "vibe coding" — the casual, prompt-driven coding style enabled by strong models
- Simon Willison's agentic engineering patterns — practical techniques for working with coding agents

What distinguishes "Software After Software" is its **economic and organizational framing** — it's less about coding technique and more about industry structure, business models, and institutional strategy.
