# Vibe Coding vs Agentic Engineering

## Definitions

### Vibe Coding
Coined in early 2025, "vibe coding" refers to AI-assisted programming where the user doesn't review or deeply understand the generated code. The user describes what they want in natural language, gets code back, and if it works, great — if not, they describe the problem and iterate. Key characteristics:
- No code review by the user
- User may not know how to program at all
- Quality/safety constraints are secondary to getting something working
- Best suited for personal tools where bugs only affect the user

### Agentic Engineering
Professional software engineering using AI coding agents as powerful tools while maintaining engineering rigor. Key characteristics:
- Engineer applies professional judgment about security, maintainability, operations, and performance
- Uses AI tools as amplifiers of existing expertise
- Goal is higher quality software delivered faster, not just faster delivery
- Appropriate for production systems affecting other people

## The Blurring Boundary (May 2026)

Simon Willison noted in May 2026 that these two approaches are converging in practice:

> "Weirdly though, those things have started to blur for me already, which is quite upsetting."

As coding agents become more reliable, even professional engineers find themselves not reviewing every line of AI-generated code for production systems. This creates a tension between the ideal of agentic engineering and the reality of how work actually gets done.

## Key Insights

### The Professional Accountability Question
- Human teams have professional reputations that create accountability
- AI agents cannot be held accountable for what they produce
- Every successful unreviewed deployment reinforces trust, creating risk of "normalization of deviance"

### The Evaluation Problem
- Traditional signals of code quality (commit history, test coverage, documentation) can now be generated by AI in minutes
- Real-world usage becomes a more reliable signal than superficial code quality markers

### Bottleneck Shifts
- Software development lifecycles were designed around the constraint that humans produce ~200 lines of code per day
- AI enables ~2,000 lines per day, breaking upstream and downstream processes
- Design processes optimized for "getting it right the first time" because implementation was expensive can become more iterative and experimental

## The Market Perspective

Matthew Yglesias (May 2026) expressed a common professional sentiment:

> "Five months in, I think I've decided that I don't want to vibecode — I want professionally managed software companies to use AI coding assistance to make more/better/cheaper software products that they sell to me for money."

This reflects the enterprise preference for proven, reliable software over DIY approaches, even when DIY is technically feasible.
